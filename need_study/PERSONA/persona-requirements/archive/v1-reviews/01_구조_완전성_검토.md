---

## 1. 검토 기준

사용자 요청은 다음 4가지로 분해된다.

| 요구 | 충족 여부 | 근거 |
|------|-----------|------|
| `.claude/agents/README.md` 기준으로 작성 | 충족 | 문서 상단에 기준 문서 명시, README의 planner/persona 운영 원칙을 반영 |
| 명시된 12개 페르소나 사용 | 충족 | 1장 표에 P-01~P-12로 12개 파일 모두 나열 |
| 유저들이 원하는 기능을 요구사항으로 정의 | 충족 | 3장 페르소나별 기능, 4장 PR-01~PR-14 요구사항 후보 |
| 새 폴더에 생성 | 충족 | `persona-requirements/10_페르소나_요구사항_정의.md`에 생성 |

---

## 2. 잘 된 점

- 문서가 확정 기능 목록이 아니라 후보 요구사항 정의서임을 명확히 밝혔다.
- README에 존재하지만 사용자 목록에는 없던 `sales-persona-cosmetics-counselor.md`를 제외한 점이 정확하다.
- 페르소나별 “원하는 기능”과 “거부 조건”을 함께 적어 기능 과확장을 막았다.
- 기존 제품 원칙인 “회사 시스템 대체 금지”, “자동 발송 금지”, “법적 문서 자동화 금지”를 요구사항 제외 조건으로 연결했다.

---

## 3. 보완 필요 사항

### 3.1 요구사항 후보의 판정 용어가 섞여 있다

`채택 후보`, `후보`, `이미 확정, 중요도 상향`이 함께 쓰인다. 추적성을 위해 아래 네 가지로 통일하는 편이 좋다.

| 현재 표현 | 권장 표현 |
|-----------|-----------|
| 채택 후보 | Candidate |
| 후보 | Candidate |
| 이미 확정, 중요도 상향 | Committed Enhancement |
| 제외 또는 별도 제품 후보 | Rejected / Separate Product |

### 3.2 PR-01~PR-14가 어떤 페르소나 페인포인트에서 나온 것인지 약하다

현재는 대상 페르소나만 적혀 있다. 요구사항별로 “대표 페인포인트”를 1개씩 추가하면 이후 의사결정 기록에 옮기기 쉽다.

예:

```markdown
PR-01 대표 페인포인트: 방문 직전 담당자/최근 상황을 못 봐 갓길에 차를 세우거나 카톡을 뒤진다.
```

### 3.3 Core 후보와 MVP 보강 후보가 중복된다

4장의 `PR-01~PR-08`과 5.2장의 “Phase 0~3에 넣을 만한 보강 후보”가 일부 같은 의미다. 둘 중 하나를 마스터로 두고 다른 섹션은 “추천 적용 방식”으로 바꾸는 것이 좋다.

---

## 4. 1차 수정 권장안

원본 문서를 바로 수정한다면 다음 순서가 적절하다.

1. 4장 요구사항 후보 표에 `대표 페인포인트` 컬럼을 추가한다.
2. 4장 분류 값을 `Committed Enhancement / Candidate / Deferred / Rejected / Separate Product`로 정리한다.
3. 5.2장은 요구사항 목록이 아니라 “PR-01~PR-08을 MVP에 반영하는 방식”으로 바꾼다.
4. 7장의 내부 자체 검토 섹션은 별도 리뷰 문서가 생겼으므로 유지하되, “초기 작성 시 자체 검토”라고 이름을 바꾼다.

---

## 5. 2차 검토로 넘길 쟁점

2차 검토에서는 구조가 아니라 **요구사항 추적성**을 본다.

- 12개 페르소나가 PR-01~PR-14에 골고루 반영됐는가
- 특정 페르소나의 요구가 과도하게 반영됐는가
- 보안/컴플라이언스 페르소나가 Rejected로만 소비되고 학습점이 빠지지 않았는가
- 기존 MVP 기능 F-01~F-33과 연결이 약한 요구사항은 무엇인가
