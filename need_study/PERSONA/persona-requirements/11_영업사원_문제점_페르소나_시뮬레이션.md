## 1. 시뮬레이션 결론

제시된 10개 문제는 대부분 실제 영업 현장 문제로 확인된다. v2에서는 5축이 아래 **7개 축**으로 재정리된다.

| 축 | 핵심 문제 | 강하게 드러난 persona |
|----|----------|----------------------|
| 정보 분산 | 고객 정보가 엑셀·카톡·사진·수첩·회사 시스템에 흩어짐 | 도매, 산업재, 자동차, 보험, 방문 화장품 |
| 기억 복원 | 방문 직전 이전 맥락을 바로 떠올리기 어려움 | 자동차, 산업재, 보험, SaaS, 부동산, 제약 |
| 후속 누락 | 다음 연락·만기·재구매·샘플 반응·잔금을 놓침 | 보험, 자동차, 휴대폰, 방문 화장품, 산업재, 부동산 |
| 관계 구조 | 가족·소개자·의사결정자·매장 담당자 관계가 머릿속에만 있음 | 보험, 자동차, 살롱, SaaS, 반도체, 방산, 제약 |
| 현장 입력 | 모바일에서 빠르게 입력하기 어렵고 사진·음성·오프라인 요구가 큼 | 도매, 방문 화장품, 리테일, 산업재 |
| **NEW · 정책 변경 추적** | 회사·본사 정책이 주 단위로 바뀌어 영업 멘트가 매주 갱신됨 | 휴대폰, 제약, 반도체 |
| **NEW · 보고/KPI 부담** | 매주 1~2시간 본사 보고서 작성 + 정량 KPI 압박 | H&B 리테일, 휴대폰, SaaS, 반도체 |

v2에서 분리된 문제는 다음 **9개**다.

1. 회사 시스템을 대체하면 안 되는 문제
2. 업종별 핵심 객체가 다르다는 문제
3. 보안/민감정보 저장 리스크
4. 빠른 입력 방식 부족
5. 오프라인/현장 네트워크 문제
6. 보고용 기록과 영업용 기억의 목적 차이
7. **NEW · 회사·본사 정책의 잦은 변경**
8. **NEW · 한 사람·조직이 여러 역할을 동시에 갖는 다층 트랙**
9. **NEW · 시니어 사용자에게는 글씨 크기·한 손 조작 자체가 사용 가능성의 임계점**

## 2. 기존 10개 문제 검증

### 문제 1. 고객 정보가 여러 곳에 흩어짐

**검증 결과: 매우 강함**

| persona | 시뮬레이션 결과 |
|---------|----------------|
| 도매 | 거래처 120곳 정보가 카톡·종이수첩·갤러리·회사 주문 앱에 흩어짐 |
| 산업재 | 공장 담당자·품목·설비 이슈·경쟁사 정보가 ERP와 개인 메모에 분리됨 |
| 자동차 | 고객 가족 정보·차량 선호·시승 이력·출고 후 관리가 본사 시스템 밖에 있음 |
| 보험 | 가족관계·생애 이벤트·보장 점검 메모가 회사 시스템과 개인 노트에 분리됨 |
| 부동산 | 고객 조건·매물 사진·임장 메모·계약 단계가 따로 관리됨 |
| **NEW · 보험·방판 (행동)** | **카톡 300+ 1:1 채팅방에 흩어진 약속·대화·메모 → 1년 전 대화 검색 거의 불가** |
| **NEW · 자동차** | **인스타·블로그·유튜브 DM으로 들어온 신규 고객 출처가 별도 추적 안 됨** |

**문제 확정**
이 문제는 가장 강한 공통 페인포인트다. 단순 고객 DB가 아니라 `카드 + 메모 + 사진 + 액션 + 카톡 대화 검색`을 한곳에 묶는 문제가 핵심이다.

### 문제 2. 방문 직전 고객 맥락을 빨리 떠올리기 어려움

**검증 결과: 매우 강함**

| persona | 시뮬레이션 결과 |
|---------|----------------|
| 자동차 | 전시장 재방문 고객이 오면 이전 관심 차종·가족 상황·가격 반응을 즉시 봐야 함 |
| 산업재 | 공장 방문 중 이전 설비 문제·구매 담당 성향·납기 민감도를 바로 확인해야 함 |
| 보험 | 상담 전 가족 구성·이전 거절 사유·보장 점검 포인트를 떠올려야 함 |
| SaaS | Zoom 미팅 직후 또는 다음 미팅 전 Champion·EB·DM 맥락을 봐야 함 |
| 부동산 | 고객이 본 매물과 탈락 사유를 기억해야 신뢰를 잃지 않음 |
| **NEW · 제약 MR** | **의사 진료실 입장 전 5초~5분 안에 처방 성향·관심 임상·마지막 디테일링 떠올림** |
| **NEW · 보험 (행동)** | **미팅 5분 전 카톡 채팅방 스크롤하면서 벼락치기 — 매번 손실 시간** |
| **NEW · SaaS (정량)** | **줌 미팅 직후 노트 정리에 1시간 (Gong + Notion + Salesforce 이중 입력)** |

**문제 확정**
이 문제는 제품의 핵심 가치와 직접 연결된다. "10초 안에 카드 조회"가 반드시 MVP 목표가 되어야 한다.

### 문제 3. 후속 연락을 놓침

**검증 결과: 매우 강함**

| persona | 놓치는 후속 |
|---------|-------------|
| 자동차 | 시승 후 연락·출고 후 사후 관리·**5~7년** 재구매 연락 |
| 보험 | 갱신/만기·보장 점검·생애 이벤트 연락 |
| 휴대폰 | 약정/할부 만료·기기 변경 가능 시점 |
| 방문 화장품 | 샘플 반응 확인·제품 소진 시점·생일/명절 선물 제안 |
| 산업재 | 견적 후 미회신·샘플 테스트 결과·교체 주기 |
| 제약 | 요청 자료 전달·다음 방문 일정 |
| **NEW · 부동산** | **임장 → 가계약 → 계약 → 잔금 → 등기 5단계, 잔금 시점 임차인 퇴거·대출 실행 동시 진행** |
| **NEW · 자동차** | **출고 일정 3중 매칭: 차량 입고 시점 + 고객 일정 + 등록·인도일** |
| **NEW · 제약 (컴플라이언스)** | **공정거래 한도(의사당 연간·식사 금액) 80% 도달 시점 알림** — 위반 시 형사 책임 |

**문제 확정**
`다음 액션`과 `리마인더`는 부가 기능이 아니라 핵심 기능이다. 다만 자동 발송은 금지하고, 영업자가 직접 연락하도록 돕는 구조가 맞다.

### 문제 5. 재구매/만기/소진 타이밍을 놓침

**검증 결과: 매우 강함 — 단, 주기 격차 크다**

| persona | 반복 타이밍 | 주기 |
|---------|-------------|------|
| 자동차 | 출고 후 재구매·리스/할부 만기 | **5~7년** |
| 보험 | 갱신·만기·보장 점검·생애 이벤트 | 1~3년 |
| 휴대폰 | 약정 만료·할부 종료·가족 결합 변경 | 2년 |
| 방문 화장품 | 크림/세럼/클렌징 소진 시점 | **월 1~2회** (가장 짧음) |
| 살롱 | 제품 소진과 재주문 타이밍 | 1~3개월 |
| 산업재 | 소모품/부품 교체 주기 | 산업별 다양 |
| 도매 | 주문 패턴과 미수금 회수 시점 | 주~월 |

**문제 확정**

### 문제 8. 보고와 실제 영업 기록이 따로 놂

**검증 결과: B2B는 매우 강함, B2C는 약함**

| persona | 분리되는 이유 | 강도 |
|---------|--------------|------|
| SaaS | Salesforce는 공식 파이프라인, 개인 앱은 비공식 맥락 | ★★★★★ |
| 반도체 | 본사 보고 양식(한글/영문 이중)과 현장 관계 메모가 다름 | ★★★★★ |
| 제약 | VEEVA/회사 CRM과 방문 준비용 메모가 다름 | ★★★★★ |
| 자동차 | 본사 DMS는 견적/계약/출고 중심·개인 고객관리는 별도 | ★★★★ |
| 산업재 | ERP는 주문/정산 중심·영업 관계 메모는 별도 | ★★★★ |
| 도매 | 회사 주문 앱은 주문만 되고 사장님 성향/미수 메모는 안 됨 | ★★★ |
| **NEW · 방판 화장품** | **본사 카운슬러 앱은 주문·적립용·단골 가족 메모는 안 됨** | ★★★ |
| 보험 (B2C) | 회사 시스템 데이터 자동 추출 거부 — 분리는 강하지만 보고 부담은 약함 | ★★ |

**문제 확정**
제품은 회사 시스템 대체가 아니라 `영업 기억 보조 도구`로 포지셔닝해야 한다. **단, B2C(보험·방판)은 보고 부담보다 단골 관계 관리가 핵심.**

## 3. 추가로 발견된 문제

### 추가 문제 A. 업종별 핵심 객체가 서로 다름

기존 10개 문제는 `고객 정보` 중심으로 표현되어 있지만, 실제 persona는 관리해야 할 객체가 다르다.

| 업종 | 핵심 객체 |
|------|-----------|
| 자동차 | 고객·차량·시승·출고·소개 |
| 보험 | 고객·가족·계약·생애 이벤트 |
| 휴대폰 | 고객·기기·약정·가족 결합 |
| 부동산 | 고객 조건·매물·임장·계약 단계 + **매수·매도·임차·임대 4트랙** |
| 리테일 | 매장·담당자·진열 사진·품절 이슈 |
| 산업재 | 공장·부서·담당자·품목·설비 |
| 제약 | 병의원·의료진·제품 반응·**약사·구매팀 별도 트랙** |
| SaaS/반도체 | 계정·의사결정자·프로젝트 단계·**POC 액션 아이템** |
| 살롱 | 매장·**인물 관계도(사장·원장·실장·디자이너) + 결정권자 표시** |
| 방문 화장품 | 단골·가족·피부타입·샘플 |

**결론**
단일 `customers/deals` 모델만으로는 부족하다. 카드 기반 모델과 업종별 템플릿이 필요하다.

### 추가 문제 B. 보안/민감정보 리스크

기존 10개 문제에는 보안이 독립 문제로 충분히 드러나지 않았다.

**강하게 나타난 persona**
- 방산: 일반 클라우드 SaaS 사용 불가
- 반도체: 보안 인증(SOC2·국정원) 없으면 회사 승인 불가
- 제약: 환자 정보·리베이트성 메모 금지
- 보험: 건강/가족/재정 정보 민감
- 휴대폰: 신분증·주민번호·계좌번호 저장 금지

**결론**
민감정보 저장 경고·민감 메모 잠금·Export 제한은 필수다. 방산은 **별도 보안형 제품**으로 분리 (7장 부록 참조).

### 추가 문제 C. 빠른 입력 방식 부족

정보를 찾는 문제만큼 중요한 것은 입력 문제다.

**예시**
- 도매: 운전 중 타이핑 불가·음성 메모 필요
- 방문 화장품: 상담 중 오래 입력 불가·큰 글씨/단순 입력 필요
- 산업재: 공장 현장에서 길게 작성 불가·운전 중 블루투스 음성
- 자동차: 전시장 응대 사이 짧게 입력해야 함
- 제약: 진료실 5분 안에는 거의 입력 불가·나오면서 한 줄
- SaaS: 줌 미팅 직후 음성 메모 → 어카운트 자동 매칭

**결론**
빠른 메모·태그 원터치·**음성 메모·사진 첨부·임시 저장**이 필요하다. v1에선 P2 위치였지만 일부 페르소나에겐 P1 임계점.

### 추가 문제 D. 오프라인/현장 네트워크 문제

특히 도매·산업재·반도체·부동산·제약(병원 안)에서 나타난다.

**상황**

**결론**
최소한 최근 카드와 최근 메모는 오프라인 조회가 가능해야 한다.

### 추가 문제 E. Export/OCR 같은 기능은 업종별로 가치가 다름

기존 기획에서 OCR/PDF Export가 중요 기능처럼 보일 수 있지만, persona별로 반응이 갈린다.

| 기능 | 가치가 높은 persona | 가치가 낮은 persona |
|------|--------------------|--------------------|
| 명함 OCR | 산업재·SaaS 일부·법인 영업 | 방문 화장품·보험·휴대폰·자동차 일부 |
| 사진 첨부 | 리테일·부동산·산업재·도매 | SaaS·보험 일부 |
| PDF Export | 부동산 제안서·리테일 보고서 | 자동차·방문 화장품·도매 |
| Excel Import | 산업재·리테일·도매 | 모바일 중심 개인 영업 일부 |

**결론**
기능 우선순위는 공통 MVP와 업종별 확장으로 분리해야 한다.

### 추가 문제 F (v2 신규). 회사·본사 정책의 잦은 변경

특히 휴대폰 매장·제약 MR·반도체 영업에서 강하게 나타난다.

**예시**
- 휴대폰: 통신사 요금제·약정·결합 조건·공시지원금이 **주 단위로 변경** → 매장 매니저가 매주 외움 + 직원 교육
- 제약: 임상 데이터·처방 가이드라인·공정거래 한도 분기별 갱신
- 반도체: 본사 forecast 기준·할인 정책·기술 인증 기준 분기별 갱신

**페인포인트**
- 본사 정책 카톡방 메시지 폭주 → 핵심 변경사항 추출 어려움
- 직원에게 매주 새로 교육해야 함 (휴대폰 매장)
- 잘못 안내 시 본사 감사·고객 신뢰 모두 깎임

**결론**
"본사 정책 주간 변경 자동 요약 + 직원 교육용 1장 카드"는 P1 후보. 단, 업종별 정책 구조가 달라 MVP 범위 과다.

### 추가 문제 G (v2 신규). 정량 KPI / 보고서 작성 부담

**강하게 나타난 persona**
- H&B 리테일: 매대 사진 + 진열 점유율 + 발주 추정 매주 본사 보고 (1~2시간)
- 휴대폰: 가입 건수·부가서비스 비율·재계약률 매주 보고
- SaaS: Salesforce 입력 의무 + 분기 Quota 압박
- 반도체: 본사 영문 주간 보고서 (한글·영문 이중 작성)

**페인포인트**
- 매주 1~2시간 정리 시간 손실
- 동일 정보 한글·영문 또는 회사·본사 2번 입력
- 미달 시 페널티

**결론**
"주간 보고서 자동 초안"은 P1~P2 후보. 회사 보고 양식이 다양해 80% 완성도 확보가 어려움.

### 추가 문제 H (v2 신규). 한 사람·조직이 여러 역할을 동시에 갖는 다층 트랙

**강하게 나타난 persona**
- 부동산: 한 사람이 매수자이자 매도자(사다리식) — **매수·매도·임차·임대 4트랙**
- 제약: 의사·약사·간호사·구매팀이 각자 별도 의사결정 트랙
- SaaS: Champion·Economic Buyer·Decision Maker·User·Blocker가 같은 어카운트 내 분리
- 반도체: 구매·기술 엔지니어·임원·해외 본사가 한 어카운트 내 다층
- 살롱: 사장·원장·실장·디자이너의 결정권 분포

**페인포인트**
- 한 카드 = 한 역할 모델로는 부족 → 한 사람이 여러 역할 매핑 필요
- 역할별 의사결정권 표시 필요

**결론**
`relationships` 모델 + 역할 다중 태깅이 필요. 문제 6(관계망)과 함께 P1 후보.

### 추가 문제 I (v2 신규). 시니어 사용자에게 글씨 크기·한 손 조작은 사용 가능성의 임계점

v1에선 문제 9·추가 C에 묻혀 있던 항목.

**강하게 나타난 persona**
- 방문 화장품 카운슬러 (47세): **노안 시작·큰 글씨/큰 버튼 필수**·작은 글씨면 즉시 안 씀
- 산업재 외근 (40세): 운전 중 한 손·시야 분산 위험
- 도매 (30세, 트럭 운행 중): 양손 자유 불가
- 방산 (44세): 보수적·작은 글씨·복잡 UI 거부

**페인포인트**
- 기본 16pt vs 시니어 20pt 모드 차이가 사용 가능성을 결정
- "한 손 조작 / 음성 메모"가 필수 (선택 옵션 아님)

**결론**
v1에선 문제 9 모바일·추가 C 빠른 입력에 묻혀 P2로 갔지만, 일부 페르소나에겐 **P0 임계점**. 6장 업종별 매트릭스에서 분리.

## 5. 최종 문제 정의

sales-persona 시뮬레이션 결과, 이 서비스가 해결해야 할 핵심 문제는 다음 한 문장으로 정리된다.

> 영업 사원은 고객과 거래처를 더 많이 기록하고 싶은 것이 아니라, 흩어진 현장 정보를 필요한 순간에 바로 떠올리고, 다음 행동을 놓치지 않으며, 회사 시스템에 적기 어려운 관계 맥락을 안전하게 관리하고 싶다.

따라서 제품의 핵심 문제는 `CRM 부재`가 아니다.
핵심 문제는 **현장 기억의 단절과 후속 행동 누락**이다.

**v2 보강**: 이 문제는 산업·연령에 따라 양상이 다르다. 같은 P 등급 문제라도 페르소나별 강도 차이가 크므로, 6장 업종별 매트릭스로 1차 타겟의 우선순위를 잡는다.

## 7. 부록 — 방산 별도 제품 검토 (v2 신규)

방산 영업(P-05 임철수)은 **일반 B2C 영업 관리 앱의 직접 타겟이 아니다.** 다만 시장 자체는 무시할 수 없으므로 다음과 같이 분리 검토 권장.

### 7.1 분리 이유

| 일반 페르소나 | 방산 페르소나 |
|--------------|--------------|
| 모바일 + 클라우드 SaaS | **외부 SaaS 자체가 보안 위반** |
| 카톡·인스타·줌 연동 | 보안망 PC + 보안 메일 + 종이 |
| 월 1만~3만 원 자비 결제 | 회사 결재 + 보안팀 승인 (6개월~1년) |
| 영업 사이클 즉시~3년 | 5~10년 |
| 단가 수백만~수억 | 수백억~수조 |

### 7.2 별도 제품 컨셉 후보

- 사내망 온프레미스 + CC 인증 + 국정원 보안적합성
- 입찰 일정·평가 기준·자격 요건 추적 (방위사업청 연동)
- 5~10년 사업 단계 (소요결정·ROC·입찰·평가·계약·양산·정비)
- 이해관계자 관계망 + 인사 이동 수동 기록
- 감사 대응용 변경 로그

### 7.3 현재 판단

이 부록은 **참고용**이며 현 MVP 로드맵에는 포함하지 않는다. 별도 B2B 보안 제품으로 검토하려면 시장 조사·기술 파트너·자본 투입이 별도로 필요.
