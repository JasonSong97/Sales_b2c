## 1. 언제 어떤 에이전트를 부를까

### 1.1 web-app-planner를 부르는 순간
- "이 기능 넣어야 할까요?" — 채택/보류/제외 분류 요청
- "MVP 범위 잡아주세요" — 기능 우선순위
- "PRD 초안 써주세요" — 문서 작성
- "User Flow 그려주세요" — 화면·행동 흐름
- "PMF 가설 세워주세요" — 검증 계획
- "인터뷰 가이드 만들어주세요" — 질문지 설계
- "이 의사결정을 기록해주세요" — Decision Log
- "경쟁 서비스 비교해주세요" — WebFetch/WebSearch로 조사

### 1.2 페르소나 3종을 부르는 순간
- **기능 검증**: "이 기능 실제로 쓸까요?" — 세 명 모두 또는 해당 산업만
- **플로우 검증**: "이 화면 흐름이 현장에서 막힐 곳 있을까요?"
- **카피·가격 검증**: "월 9,900원 vs 1년 99,000원, 어느 게 끌려요?"
- **인터뷰 리허설**: 실제 사용자 인터뷰 전 질문지 다듬기
- **반증 신호 수집**: "이 가설을 깨는 답이 나올 수 있는지" 확인

### 1.3 어떤 페르소나를 부를지

| 검증 포인트 | 적합한 페르소나 |

## 2. 호출 방법

### 2.1 Task 도구로 호출 (메인 세션에서)

```
Task(
  subagent_type="web-app-planner",
  description="음성 메모 기능 분류",
  prompt="음성 메모 → 거래처 자동 매칭 기능을 확정/후보/보류/제외 중 어디로 분류해야 하는지 판단해주세요. 07번과 04번 문서를 먼저 보고 판단해주세요."
)
```

페르소나 6종은 산업이 다르므로 **병렬 호출**이 권장됩니다. 다만 한 번에 6종 다 부를 필요는 없고, **검증 대상 기능과 관련된 페르소나 2~4종**을 골라서 병렬 호출합니다.

```
Task(subagent_type="sales-persona-industrial",          prompt="...")
Task(subagent_type="sales-persona-insurance",            prompt="...")
Task(subagent_type="sales-persona-cosmetics-counselor",  prompt="...")
```

기능 성격에 따라 어떤 페르소나 조합이 좋은지는 1.3 절 매트릭스와 1.4 절 산업별 매핑을 참고하세요.

### 2.2 프롬프트 작성 원칙

- **목적을 먼저** 말한다: "분류 판단" / "기획 피드백" / "인터뷰 시뮬레이션"
- **검증할 가설/기능을 명시**한다 (F-## 또는 한 줄 설명)
- 페르소나 호출 시 **모드를 명시**한다: "기획 피드백 모드" / "인터뷰 시뮬레이션 모드"
- 길게 답해야 할 때는 **명시적으로** 요청 ("5문장 이하로" 또는 "10분 분량으로")
- 인터뷰 시뮬레이션에서는 **인터뷰어 역할을 직접** 한다 ("제가 인터뷰어, 당신이 응답자")

## 3. 웹/앱 서비스 만들기 — 시나리오 모음

서비스를 **0에서 베타·확장까지** 만들 때 페르소나·기획자 에이전트를 어떻게 조합해 쓰는지 단계별로 정리. 각 시나리오는 **상황 → 목표 → 호출 순서 → 산출물 → 프롬프트 예시**로 구성.

### 3.1 시나리오 인덱스

| Stage | 시나리오 | 주 에이전트 | 산출물 |
|-------|---------|------------|--------|
| 0 | 컨셉 정제 (아이디어 → PRD 1장) | planner | PRD 초안, 가설 5개 |
| 1 | 타겟 페르소나 선정 | planner + 페르소나 5~7명 | 1차 타겟 1~2명, 페인 강도 점수 |
| 2 | MVP 기능 범위 결정 | planner + 페르소나 3~5명 | 확정 기능 4~6개, 결정 기록 |
| 3 | User Flow 설계 + 검증 | planner + 페르소나 1~3명 | Flow 5종, 막히는 지점 |
| 4 | 인터뷰 가이드 + 리허설 | planner + 페르소나 1~3명 | 인터뷰 가이드 v2 |
| 5 | 가격·결제 모델 검증 | planner + 페르소나 3~5명 | 가격 정책 결정 기록 |
| 6 | 카피·네이밍·랜딩 검증 | 페르소나 3~5명 | 헤드라인 + A/B 변형 |
| 7 | 에러·예외 케이스 검증 | planner + 페르소나 2~3명 | 에러 UX 정책, 신뢰 임계점 |
| 8 | 베타 피드백 분석 | planner (+ 페르소나 선택적) | 다음 분기 백로그 |
| 9 | 산업 확장 결정 | planner + 후보 산업 페르소나 2~3명 | 확장 의사결정 기록 |

각 시나리오는 독립적으로 쓸 수 있고, 0→9 순차 실행도 가능. **빠르게 쓰고 싶다면 3.13 단발성 패턴**부터 보세요.

### 3.2 [Stage 0] 컨셉 정제 — 아이디어 한 줄을 PRD로

**상황**: 한 줄 컨셉이 있다. "X 산업 영업사원을 위한 Y 도구" 같은 형태.

**목표**: 한 줄을 PRD 1장 + 검증 가능한 PMF 가설 5개로 분해.

**호출 순서**:
1. `web-app-planner` — 컨셉 입력 → (a) 타겟 사용자 (b) 핵심 페인 3개 (c) 현재 대체재 (d) 차별 가치 (e) PMF 가설 5개 출력
2. 가설 5개 중 **가장 강한 2개**를 검증 대상으로 지정
3. 가설별로 검증에 적합한 페르소나 1~2명 매핑

**산출물**: PRD 1장 / PMF 가설 5개(H-01~H-05) / 검증 대상 페르소나 매핑

**프롬프트**:
```
Task(subagent_type="web-app-planner",
     prompt="컨셉 1줄: 'B2B 산업재 영업사원이 운전 중 음성으로 거래처 비공식 메모를
            남기고 사무실에서 다시 듣는 도구'.
            이걸 PRD 1장 분량으로 정리 + PMF 가설 5개(H-01~H-05) 작성 +
            각 가설을 검증할 가장 적합한 페르소나 매핑까지 부탁드립니다.")
```

### 3.3 [Stage 1] 타겟 페르소나 선정 — "누가 진짜 고객인가?"

**상황**: 컨셉이 여러 산업에 적용 가능해 보임. "운전 중 메모"는 산업재·도매·부동산·자동차·제약 모두 해당.

**목표**: 13종 중 **누구의 페인이 가장 강한지** 비교 → 1차 타겟 1~2명에 집중.

**호출 순서**:
1. 관련 산업 페르소나 5~7명 **병렬** 호출
2. **동일 질문**: "이런 도구가 있다면 한 달에 몇 번 쓸 것 같아요? 안 쓸 이유 1~2개도 솔직히"
3. 응답 비교 — **빈도 + 강도 + 반증 신호** 3축 점수화
4. 점수 1~2위를 **1차 타겟**으로 확정

**산출물**: 페르소나별 페인 강도 점수표 / 1차 타겟 1~2명 + 사용 빈도 추정

**프롬프트** (병렬 5개):
```

     prompt="기획 피드백 모드. 운전 중 음성으로 거래처 비공식 메모를 남기고 사무실에서
            다시 듣는 도구. 한 달에 몇 번 쓸 것 같아요? 안 쓸 이유 1~2개 솔직히.")
Task(subagent_type="sales-persona-wholesale",  prompt="<동일>")
Task(subagent_type="sales-persona-realestate", prompt="<동일>")
Task(subagent_type="sales-persona-automotive", prompt="<동일>")
```

### 3.4 [Stage 2] MVP 기능 범위 — 후보 20개 → 확정 5개

**상황**: 1차 타겟 정해짐. 4주 안에 무엇을 만들지 결정.

**목표**: 기능 후보 20개를 **확정 / 후보 / 보류 / 제외** 4분류 → MVP = "확정"만 4~6개.

**호출 순서**:
1. `web-app-planner` — 후보 20개를 한 줄씩 → 4분류 초안 + 분류 근거
2. "**후보 (Candidate)**" 분류된 기능 각각에 1차 타겟 페르소나 검증 (병렬)
3. `web-app-planner` — 페르소나 응답 종합 → 최종 분류 + 결정 기록 D-001~
4. MVP 범위 = "확정"만

**산출물**: 기능 명세 (F-01~) / 결정 기록 (D-001~) / Out of Scope 리스트

**핵심 원칙**:
- MVP는 **5개 이하**로 시작
- "후보"는 출시 후 베타 데이터 보고 재판단
- 통과 기준: **"이 기능 빼면 핵심 경험이 깨지는가?"**

**프롬프트 (1단계)**:
```
Task(subagent_type="web-app-planner",
     prompt="기능 후보 20개 (아래 목록). 각각을 '확정/후보/보류/제외'로 분류 +
            분류 근거 한 줄 + MVP(확정만) 5개 선정 + 출시 후 검토할 '후보' 따로 정리.

            1. 음성 메모 자동 거래처 매칭
            2. 거래처 카드 (가족·취향 메모)
            3. 명함 OCR
            4. 오프라인 동기화
            5. 검색·필터링
            ...
            20. 견적서 자동 생성")
```

### 3.5 [Stage 3] User Flow 설계 + 시뮬레이션 검증

**상황**: MVP 5개 확정. 화면 흐름 그리는 단계.

**목표**: 페르소나가 **"현장에서 막힐 곳"**을 발견하게 한다.

**호출 순서**:

2. 1차 타겟 페르소나에 **시점 시뮬레이션**: "Flow-A를 토요일 오후 운전 중에 쓴다고 가정하고 한 단계씩 뭘 할지 묘사해주세요"
3. 페르소나가 "여기서 헷갈리겠다", "이 화면 한 손으로 못 누른다" 같은 지점을 자가 표시
4. `web-app-planner` — Flow 수정 + 예외 경로 추가

**산출물**: User Flow 5종 / 막히는 지점 + 수정안 / 와이어프레임 핵심 화면 목록

**프롬프트**:

```
Task(subagent_type="sales-persona-industrial",
```

### 3.6 [Stage 4] 인터뷰 가이드 + 가상 리허설

**목표**: 답을 유도하지 않고 진짜 행동·페인을 끌어내는 가이드.

**호출 순서**:
1. `web-app-planner` — 가설 H-01 검증용 인터뷰 가이드 30분 분량 초안
2. 1차 타겟 페르소나로 가상 인터뷰 (인터뷰어 = 호출자, 응답자 = 페르소나)
3. 페르소나가 "이 질문은 유도예요" 자가 지적 — 인터뷰 시뮬레이션 모드가 그렇게 설계됨
4. `web-app-planner` — 가이드 수정 (유도 제거 + 정량 질문 + 반증 신호 유도)
5. 다른 산업 페르소나로 한 번 더 리허설 → 답이 갈리는지 확인

**산출물**: 인터뷰 가이드 v1 → v2 / 5명용 빈 응답표

**프롬프트**:
```

            현재 행동 → 대체 도구 → 페인포인트 정량화 → 컨셉 반응 순서로.")

Task(subagent_type="sales-persona-industrial",
     prompt="인터뷰 시뮬레이션 모드. 제가 인터뷰어, 차장님이 응답자입니다.
```

3. 2~3개 추려서 다시 검증
4. `web-app-planner` — 최종 카피 + A/B 변형

**산출물**: 헤드라인 1개 + 서브헤드라인 1개 / A/B 테스트 변형 1~2개


## 4. 잘 쓰는 팁

### 4.1 페르소나는 친구 아니다 — 호의적인 답을 경계한다
페르소나 3종은 **반증 신호도 솔직히 말하도록** 설계되어 있지만, 질문이 유도형이면 호의적으로 답할 수 있습니다.
- ❌ "이 기능 좋아 보이지 않으세요?"
- ✅ "이 기능, 안 쓰게 되는 상황이 있다면 어떤 경우일까요?"

### 4.2 페르소나에게 "기획자처럼 답하라"고 시키지 않는다
페르소나는 의사결정자가 아닙니다. "그래서 만들어야 해요 말아야 해요?"는 web-app-planner에게 물어야 합니다.
페르소나는 **현장 시점의 입력**만 주고, **종합 판단은 기획자**가 합니다.

### 4.3 기획자에게 "페르소나처럼 답하라"고 시키지 않는다
web-app-planner는 분류·문서화 전문가입니다. 현장 톤이 필요하면 페르소나를 부르세요.

### 4.4 인터뷰 시뮬레이션은 실제 인터뷰의 30%만 대체한다
페르소나는 잘 설계된 가상 사용자지만, **진짜 사용자가 아닙니다.**
- 시뮬레이션 목적: 질문지 다듬기, 명백히 빠진 페인포인트 발견, 가설 사전 점검
- 시뮬레이션이 못 하는 것: 진짜 페인포인트 발견 (모르는 걸 모르도록 설계됨), 정량 데이터 검증

### 4.5 답이 갈리면 그게 정보다
세 페르소나 답이 일치 → 기능 방향 확실
답이 갈림 → 산업별 분리 또는 범위 축소가 필요한 신호

### 4.6 페르소나 답을 그대로 결정 기록에 옮기지 않는다
페르소나 답은 web-app-planner의 입력입니다. 결정 기록(`D-###`)은 web-app-planner가 종합해서 씁니다.

### 4.7 산업 컨텍스트가 다르면 페르소나 산업을 지정한다
박상호는 기본값이 "자동차 부품"이지만 프롬프트에서 "철강 도매로 가정하고 답해주세요"라고 하면 그쪽으로 답합니다.

## 5. 흔한 실수 (안티 패턴)

### 5.1 모든 질문을 한 페르소나에만 묻기
한 산업의 답만 듣고 결정하면 다른 산업의 페인을 놓칩니다. **기본은 3종 병렬 호출**.

### 5.2 페르소나 호출 전에 web-app-planner 안 부르기
정리 안 된 아이디어를 페르소나에게 던지면 답도 정리 안 된 채로 옵니다.
**순서**: web-app-planner로 기능을 한 문장으로 정리 → 페르소나에게 질문.

### 5.3 인터뷰 시뮬레이션에서 제품 설명 먼저 하기
페르소나는 **제품 설명 전에 현재 행동부터 묻도록** 설계되어 있습니다.
- ❌ "이런 앱이 있는데 어떻게 보세요?" → 호의적 답이 옴
- ✅ "지금 거래처 메모는 어떻게 관리하세요?" → 진짜 행동이 나옴

### 5.4 페르소나 답을 "사용자 데이터"로 인용
페르소나는 **가상 인물**입니다. PMF 검증의 근거로 쓰면 안 되고, **가설 사전 점검·인터뷰 리허설용**입니다.
PMF 근거는 진짜 인터뷰 / 실사용 로그 / 베타 신청자 데이터에서만 나옵니다.

### 5.5 한 번에 5개 이상 질문 던지기
페르소나는 사람이 답하듯 답합니다. 한 번에 너무 많이 물으면 답이 얕아집니다.
**한 호출당 핵심 질문 1~3개**가 적당합니다.

### 5.6 페르소나에게 코드·DB·기술 스택 묻기
범위 밖입니다. 페르소나는 **현장 사용자**, web-app-planner는 **기획자**, 코드는 다른 영역(Frontend/Backend/Mobile AGENT 폴더의 규칙)에서 다룹니다.

## 7. 추가 / 수정 / 제거

- **에이전트 수정**: 해당 `.md` 파일의 frontmatter(`description`, `tools`)와 본문을 편집합니다. 다음 호출부터 반영됩니다.
- **새 페르소나 추가**: 같은 형식으로 `sales-persona-{type}.md` 생성. 기존 3종과 **페인포인트가 겹치지 않게** 차별화 포인트를 명확히 잡습니다.
- **활성 에이전트 확인**: 메인 세션에서 `/agents` 명령으로 목록 확인.
