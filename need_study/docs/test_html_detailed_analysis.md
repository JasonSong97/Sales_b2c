## 2. 제품 해석

### 2.1 제품 성격

SalesKit은 B2B 영업 파이프라인을 관리하는 업무형 CRM 프로토타입이다. 현재 파일은 완성 제품이 아니라, 산업 장비 영업 흐름을 설명하기 위한 프론트엔드 데모에 가깝다.

핵심 도메인은 다음 네 가지다.

| 도메인 | 의미 | 현재 구현 |
| --- | --- | --- |
| 딜 | 고객사에 제안 중인 영업 기회 | 목록, 필터, 정렬, 상세 패널, 활동 기록 |
| 거래처 | 고객사 담당자 | 카드 목록, 검색, OCR 시뮬레이션 |
| 회사 | 고객사 마스터 | 카드 목록, 회사 등록 모달 |
| 제품 | 제안 대상 제품과 단가 | 테이블 목록, 단가 이력 표시, 등록 모달 |

### 2.2 주요 업무 시나리오

1. 영업 담당자가 오늘 관리해야 할 딜을 확인한다.
2. 딜 목록에서 지역, 확도, 마감일 기준으로 우선순위를 좁힌다.
3. 특정 딜을 클릭해 상세 패널을 연다.
4. 고객사, 담당자, 제품, 금액, 확도, 마감일, 활동 기록, 메모를 확인한다.
5. 미팅이나 통화 결과를 활동 기록에 추가한다.
6. 신규 딜 등록 모달에서 제품을 입력하고 단가 자동 입력을 확인한다.
7. 거래처, 회사, 제품 정보를 별도 메뉴에서 조회한다.
8. 설정 화면에서 알림, 휴지통, 데이터 관리 영역을 확인한다.

### 2.3 데모와 실제 제품의 차이

현재 구현은 브라우저 메모리 안에서만 동작한다. 데이터 저장, 권한, API, 검증, 파일 업로드, 실제 OCR, 실제 엑셀/PDF 내보내기는 구현되어 있지 않다.

| 기능 | 현재 상태 | 실제 제품화 시 필요한 보강 |
| --- | --- | --- |
| 데이터 조회 | JS 배열을 화면에 렌더링 | API 조회, 페이지네이션, 권한 필터 |
| 데이터 등록 | 등록 버튼 클릭 시 alert만 표시 | 입력값 검증, API 저장, 목록 갱신 |
| 활동 기록 추가 | 현재 세션의 `DEALS` 배열에만 추가 | 서버 저장, 작성자, 생성일, 수정/삭제 |
| 내보내기 | alert만 표시 | CSV/XLSX/PDF 생성, 필터 조건 반영 |
| OCR | 고정 예시값 표시 | 이미지 업로드, OCR API, 필드 신뢰도 |
| 알림 | 토글 UI만 있음 | 사용자 설정 저장, 스케줄러, 이메일/푸시 발송 |

## 3. 화면 정보 구조

### 3.1 최상위 레이아웃

최상위는 `.app` 컨테이너이며, 좌측 사이드바, 중앙 메인 영역, 우측 상세 패널을 수평으로 배치한다.

| 영역 | CSS/DOM | 역할 |
| --- | --- | --- |
| 사이드바 | `.sidebar` | 브랜드, 페이지 이동, 내보내기 진입, 사용자 정보 |
| 메인 | `.main` | 상단바, 탭, 필터, 각 페이지 콘텐츠 |
| 상세 패널 | `.detail-panel` | 딜 행 선택 시 우측에서 상세 정보 표시 |

`.app`은 `height:100vh`와 `overflow:hidden`으로 브라우저 높이에 맞춘 업무 앱 형태를 만든다. 스크롤은 전체 페이지가 아니라 `.content-area`, `.sidebar-nav`, `.detail-body`, `.modal-body` 같은 내부 영역에만 생긴다.

### 3.2 페이지 목록

`goPage(page)` 함수가 페이지 전환을 담당한다. 전환 가능한 페이지는 다음 배열로 고정되어 있다.

| page 값 | DOM ID | 메뉴 ID | 화면명 | 표시 방식 |
| --- | --- | --- | --- | --- |
| `dashboard` | `page-dashboard` | `nav-dashboard` | 대시보드 | `block` |
| `deals` | `page-deals` | `nav-deals` | 딜 관리 | `flex` |
| `contacts` | `page-contacts` | `nav-contacts` | 거래처 | `flex` |
| `companies` | `page-companies` | `nav-companies` | 회사 | `flex` |
| `products` | `page-products` | `nav-products` | 제품 | `flex` |
| `settings` | `page-settings` | `nav-settings` | 설정 | `block` |

페이지 이동 시 모든 페이지는 먼저 `display:none` 처리된다. 이후 선택된 페이지만 `flex` 또는 `block`으로 표시된다. 동시에 사이드바의 active 클래스, 상단 제목, 상단 액션 버튼이 갱신된다.

### 3.3 공통 상단바

상단바는 `#page-title`과 `#topbar-actions`로 구성된다. `goPage()`는 페이지별로 다음 액션을 주입한다.

| 페이지 | 상단 액션 |
| --- | --- |
| 딜 관리 | 내보내기, 새 딜 등록 |
| 거래처 | 명함 스캔 |
| 회사 | 회사 등록 |
| 제품 | 제품 등록 |
| 대시보드 | 없음 |
| 설정 | 없음 |

상단 액션 HTML은 JavaScript 문자열로 직접 삽입된다. 유지보수 측면에서는 액션 정의를 템플릿 함수나 컴포넌트로 분리하는 것이 좋다.

### 3.4 사이드바

사이드바는 세 구획으로 구성된다.

| 구획 | 내용 | 현재 동작 |
| --- | --- | --- |
| 로고 | `SalesKit`, 영업 관리 도구 | 정적 텍스트 |
| 메뉴 | 대시보드, 딜 관리, 거래처, 회사, 제품, 설정 | `goPage()` 호출 |
| 내보내기 | 엑셀 내보내기, PDF 내보내기 | alert만 표시 |
| 사용자 | 김영업, 계측기사업부 | hover UI만 있음 |

메뉴의 배지는 현재 데이터에서 계산하지 않고 HTML에 고정되어 있다. 예를 들어 딜 관리 배지는 12지만 실제 `DEALS` 배열은 8건이다.

### 5.1 데이터 원천

| 상수 | 라인 | 실제 건수 | 설명 |
| --- | ---: | ---: | --- |
| `DEALS` | 611 | 8 | 영업 기회 데이터 |
| `CONTACTS` | 621 | 6 | 고객 담당자 데이터 |

| `PRICES` | 786 | 5 | 신규 딜 모달의 제품명별 자동 단가 |

### 5.2 Deal 모델

`DEALS`의 각 객체는 다음 필드를 가진다.

| 필드 | 타입 | 예시 | 설명 |
| --- | --- | --- | --- |
| `id` | number | `1` | 딜 고유 ID |
| `title` | string | `DPT-3000A 납품` | 딜명 |
| `contact` | string | `삼성전자 DX` | 화면 표시용 거래처/조직명 |
| `person` | string | `송재근` | 담당자명 |
| `company` | string | `삼성전자` | 회사명 검색용 문자열 |
| `region` | string | `수원` | 지역 필터 기준 |
| `stage` | string | `negotiation` | 단계 코드 |
| `amount` | number | `48000000` | 예상 금액, 원 단위 |
| `prob` | number | `85` | 구매 가능성, 0-100 |
| `date` | string | `2026.05.31` | 마감 예정일 |
| `product` | string | `디지털 압력 트랜스미터 DPT-3000A` | 제안 제품 |
| `memo` | string | `5월 3주차 미팅 완료...` | 일반 메모 |
| `off` | string | `DX사업부 내 실제 결정권...` | 비공개 메모 |
| `activities` | array | `{date,text,auto}` | 활동 기록 배열 |

현재 Deal은 `companyId`, `contactId`, `productId` 같은 관계형 참조를 사용하지 않는다. 대신 표시 문자열을 중복 저장한다. 데모에는 적합하지만, 실제 서비스에서는 정규화가 필요하다.

### 5.3 Stage 모델

`stage` 값은 네 가지를 의도한다.

| 코드 | 표시명 | 배지 색상 | 실제 데이터 건수 |
| --- | --- | --- | ---: |
| `prospect` | 초기접촉 | 파랑 | 3 |
| `negotiation` | 협의중 | 앰버 | 3 |
| `won` | 성사 | 초록 | 2 |
| `lost` | 실패 | 빨강 | 0 |

딜 탭에는 실패 1건으로 표시되어 있지만 실제 `DEALS`에는 `lost` 단계가 없다. 이 불일치는 UI 신뢰도를 떨어뜨리므로 배지를 데이터 기반으로 계산해야 한다.

### 5.4 Activity 모델

활동 기록은 Deal 내부 배열로 포함된다.

| 필드 | 타입 | 설명 |

| `date` | string | 활동 발생일 |
| `text` | string | 활동 내용 |
| `auto` | boolean | 단계 변경 같은 자동 기록 여부 |

`auto:true`인 기록은 `.tl-dot.auto` 스타일을 사용해 일반 수동 기록과 구분한다. 현재는 작성자, 생성 시각, 수정/삭제 권한이 없다.

### 5.8 데이터 카운트 불일치

| 협의중 | 5 | 3 | 탭 불일치 |

| 실패 | 1 | 0 | 탭 불일치 |

| 회사 | 11 | 6 | 사이드바/탭 불일치 |

이 문제는 데모 데이터가 축소되었는데 UI 문구가 갱신되지 않았거나, 더 큰 데이터셋을 상정한 정적 숫자가 남아 있기 때문에 발생한 것으로 보인다.

## 6. 화면별 상세 분석

### 6.1 대시보드

대시보드는 전체 현황을 빠르게 보여주는 요약 화면이다. 현재 모든 지표와 목록이 정적 HTML로 작성되어 있다.

#### 표시 지표

| 지표 | 화면 표시 | 현재 데이터 기준 검토 |
| --- | --- | --- |
| 전체 딜 | 12건, 진행중 9건 | 실제 딜 8건, 진행중은 prospect+negotiation 기준 6건 |
| 이번달 마감 예정 | 3건, 106,000,000원 | 실제 2026년 5월 딜은 4건 |
| 이번달 성사 금액 | 44,000,000원 | 실제 won 딜 38,000,000 + 6,000,000으로 일치 |
| 가능성 70%+ 딜 | 4건, 196,000,000원 | 실제 5건, 190,400,000원 |

#### 우선 관리 딜

정적 테이블에 3건이 표시된다.

| 딜 | 확도 | 마감 |
| --- | ---: | --- |
| DPT-3000A 납품 | 85% | 2026.05.31 |
| 가스분석기 도입 | 75% | 2026.05.28 |
| 압력계 정기 교체 | 70% | 2026.06.10 |

행 클릭 시 특정 딜 상세가 아니라 `goPage('deals')`만 호출한다. 따라서 딜 목록으로 이동할 뿐 해당 딜이 선택되지는 않는다.

#### 최근 거래처

최근 거래처 역시 정적 테이블이다. 행 클릭 시 거래처 페이지로 이동하지만 특정 거래처 상세는 열리지 않는다.

### 6.2 딜 관리 페이지

딜 관리 페이지는 현재 구현의 중심 기능이다.

#### 탭 필터

| 탭 | `dealTab` 값 | 필터 조건 |
| --- | --- | --- |
| 전체 | `all` | 전체 |
| 초기접촉 | `prospect` | `d.stage === 'prospect'` |
| 협의중 | `negotiation` | `d.stage === 'negotiation'` |
| 성사 | `won` | `d.stage === 'won'` |
| 실패 | `lost` | `d.stage === 'lost'` |
| 이번달 마감 | `thismonth` | 브라우저 현재 연월과 `d.date`의 연월 비교 |
| 가능성 70%+ | `hot` | `d.prob >= 70` |

`thismonth`는 `new Date()`를 사용하므로 브라우저 실행 시점의 로컬 시간에 의존한다. 데모 날짜가 2026년 5월에 고정되어 있기 때문에, 다른 달에 실행하면 결과가 0건이 될 수 있다.

#### 검색

검색 입력값은 소문자로 변환한 뒤 다음 필드를 합친 문자열에서 포함 여부를 검사한다.

- `title`
- `contact`
- `person`
- `company`
- `memo`

`product`, `region`, `off`, `activities.text`는 검색 대상이 아니다. 검색 placeholder에는 딜명, 거래처, 회사, 메모가 명시되어 있어 현재 동작과 거의 일치한다.

#### 필터

| 필터 | DOM ID | 값 | 동작 |
| --- | --- | --- | --- |
| 지역 | `region-filter` | 서울, 경기, 수원, 대전, 포항, 여수, 당진, 이천 | `d.region === region` |
| 가능성 | `prob-filter` | 70, 50, 30 | `d.prob >= minProb` |
| 정렬 | `sort-filter` | 날짜, 금액, 가능성 | 조건별 `Array.sort()` |

지역 옵션에는 서울, 경기처럼 현재 데이터에 없는 값도 포함되어 있다. 선택 시 빈 상태가 표시된다.

#### 정렬

| 정렬 값 | 정렬 방식 |
| --- | --- |
| `updated` | 별도 정렬 없음. 배열 원래 순서 유지 |
| `date_asc` | `date.localeCompare()` 오름차순 |
| `date_desc` | `date.localeCompare()` 내림차순 |
| `amount_desc` | 금액 내림차순 |
| `amount_asc` | 금액 오름차순 |
| `prob_desc` | 확도 내림차순 |

날짜 형식이 `YYYY.MM.DD`라서 문자열 비교가 현재 데이터에서는 정상 동작한다. 다만 날짜 타입으로 변환하지 않기 때문에 형식이 흔들리면 정렬 정확도가 깨질 수 있다.

#### 목록 렌더링

`renderDeals()`는 필터링된 딜을 `<tbody id="deal-tbody">`에 `innerHTML`로 삽입한다.

표시 컬럼은 다음과 같다.

| 컬럼 | 값 |
| --- | --- |
| 딜명 | `d.title` |
| 거래처 / 담당자 | `d.contact`, `d.person` |
| 지역 | `d.region` |
| 단계 | `STAGE[d.stage]` |
| 예상 금액 | `₩${(d.amount/10000).toFixed(0)}만` |
| 가능성 | progress bar, `d.prob` |
| 마감 예정 | `d.date` |

딜 데이터는 사용자 입력값이 아닌 하드코딩 데이터라 현재는 큰 문제가 없지만, 실제 서비스에서 `innerHTML`을 그대로 쓰면 XSS 위험이 생긴다.

### 6.3 딜 상세 패널

딜 행을 클릭하면 `openDetail(id)`가 실행된다.

#### 패널 상태

| 상태 | 구현 |
| --- | --- |
| 닫힘 | `.detail-panel.hidden` 클래스가 적용되어 width 0 |
| 열림 | hidden 클래스 제거, 310px 패널 표시 |
| 선택 행 | `openDealId`와 일치하는 행에 `selected` 클래스 적용 |
| 닫기 | `closeDetail()` 호출, `openDealId=null`, 목록 재렌더 |

#### 정보 탭

정보 탭은 다음 값을 표시한다.

- 딜명
- 거래처 / 담당자
- 제품
- 예상 금액
- 단계
- 구매 가능성
- 지역
- 마감 예정일
- 수정, 복사, 단계 변경 버튼

수정과 단계 변경 버튼은 alert만 표시한다. 복사는 `copyDeal()`에서 안내 alert만 표시하고 실제 데이터를 추가하지 않는다.

#### 활동 기록 탭

활동 기록은 `d.activities` 배열을 타임라인 형태로 보여준다. `addActivity()`는 입력값이 있으면 오늘 날짜 문자열을 만들고 `activities.unshift()`로 배열 앞에 새 기록을 추가한다.

이 동작은 현재 브라우저 세션에서는 반영되지만, 새로고침하면 사라진다. 서버 저장이나 localStorage 저장은 없다.

#### 메모/비공개 탭

일반 메모는 `d.memo`, 비공개 메모는 `d.off`를 표시한다. 메모 추가/수정 버튼은 UI만 있고 실제 편집 기능은 없다.

비공개 메모는 UI상 "나만 보임"으로 표현되지만 실제 권한이나 암호화가 없다. 모든 데이터가 HTML 소스에 그대로 존재하므로 보안 의미의 비공개가 아니다.

휴지통 모달은 삭제된 항목 3건을 정적 테이블로 보여준다. 복구 버튼은 alert만 표시한다. 실제 삭제 데이터나 복구 상태 관리는 없다.

## 7. JavaScript 동작 분석

### 7.1 전역 상태

| 변수 | 초기값 | 역할 |
| --- | --- | --- |
| `dealTab` | `all` | 현재 딜 탭 필터 |
| `openDealId` | `null` | 현재 상세 패널에 열린 딜 ID |

이 두 상태는 전역 변수다. 페이지 이동, 상세 패널, 목록 선택 상태가 모두 여기에 의존한다.

### 7.2 `renderDeals()`

`renderDeals()`는 가장 중요한 함수다.

동작 순서:

1. 검색어, 지역, 확도, 정렬 값을 DOM에서 읽는다.
2. `DEALS.filter()`로 검색, 지역, 확도, 탭 조건을 적용한다.
3. 정렬 옵션이 있으면 `data.sort()`를 수행한다.
4. 결과가 없으면 빈 상태 HTML을 삽입한다.
5. 결과가 있으면 테이블 행 HTML을 생성한다.

주의할 점:

- `sort()`는 필터링 결과 배열에 대해 수행되므로 원본 `DEALS` 배열을 직접 정렬하지는 않는다.
- `updated` 정렬은 별도 로직이 없어 원본 배열 순서다.
- 단계별 탭 카운트는 이 함수에서 갱신하지 않는다.
- `innerHTML` 기반 렌더링이라 외부 입력 데이터가 들어오면 escaping이 필요하다.

### 7.3 `openDetail(id)`

딜 상세 패널을 열고 각 DOM 노드에 값을 채운다.

주요 동작:

- `DEALS.find()`로 대상 딜 조회
- `openDealId` 갱신
- 상세 패널 hidden 제거
- 텍스트/HTML/스타일 직접 업데이트
- 활동 기록 렌더링
- 상세 탭을 정보 탭으로 초기화
- 목록을 다시 렌더링해 선택 행 표시

`STAGE[d.stage]`는 HTML 문자열이므로 `innerHTML`을 사용한다. 나머지 텍스트 값은 대부분 `textContent`로 넣고 있어 상대적으로 안전하다.

### 7.4 `addActivity()`

현재 열린 딜에 활동 기록을 추가한다.

동작 조건:

- 입력값이 비어 있으면 아무것도 하지 않는다.
- `openDealId`에 해당하는 딜이 없으면 아무것도 하지 않는다.
- 날짜는 브라우저의 현재 날짜로 생성한다.
- 추가 기록은 `auto:false`로 들어간다.

이 기능은 현재 파일에서 실제로 데이터 배열을 변경하는 거의 유일한 기능이다. 다만 영속성이 없다.

### 7.5 `goPage(page)`

페이지 전환을 수행한다.

주요 책임:

- 모든 페이지 숨김
- 모든 nav active 제거
- 선택 페이지 표시
- 선택 nav active 추가
- 제목 변경
- 상단 액션 HTML 주입
- 선택 페이지에 따라 목록 렌더링
- `closeDetail()` 호출

주의할 점은 `goPage()` 마지막에 항상 `closeDetail()`을 호출한다는 점이다. 딜 페이지로 이동하더라도 기존 상세 선택은 닫힌다. 대시보드 우선 관리 딜을 클릭해도 특정 행이 열리지 않는 이유다.

### 7.6 `openModal()` / `closeModal()`

모달은 `.modal-overlay.show` 클래스로 열고 닫는다.

추가로 모든 `.modal-overlay`에는 클릭 이벤트가 등록되어 있다. 오버레이 자체를 클릭하면 닫히고, 모달 내부 클릭은 닫히지 않는다.

`closeModal()`은 OCR 모달에 한해 step 상태를 초기화한다. 다른 모달 입력값은 초기화하지 않는다.

### 7.7 `autoFillPrice()`

신규 딜 모달의 제품 입력값이 `PRICES` 객체의 키와 일치하면 예상 금액을 자동 입력한다.

한계:

- 부분 검색이나 모델명만 입력한 경우는 동작하지 않는다.
- 제품명이 바뀌어 매칭이 없어져도 기존 금액을 지우지 않는다.
- 수량 개념이 없어 단가와 예상 금액이 동일하게 취급된다.

## 9. 품질 리스크

### 9.1 데이터 정확성 리스크

- UI 카운트가 실제 배열 데이터와 맞지 않는다.
- 대시보드 지표 일부가 실제 데이터 계산 결과와 다르다.
- 회사/거래처/제품 수가 실제 배열보다 크게 표시된다.
- `CONTACTS.deals`, `COMPANIES.deals`는 실제 `DEALS` 배열에서 계산되지 않는다.
- 날짜가 문자열로 저장되어 형식 변경 시 정렬/필터가 깨질 수 있다.

### 9.2 보안 리스크

- 데이터 렌더링에 `innerHTML`을 많이 사용한다.
- 실제 사용자 입력 데이터가 들어오면 XSS 방어가 필요하다.
- 비공개 메모가 HTML 소스 안에 평문으로 존재한다.
- 권한, 인증, 사용자별 데이터 분리가 없다.

### 9.3 접근성 리스크

- 많은 클릭 대상이 `<button>`이 아니라 `<div onclick>`이다.
- `role`, `aria-selected`, `aria-expanded`, `aria-modal` 같은 보조 속성이 없다.
- 모달 오픈 시 포커스 이동과 포커스 트랩이 없다.
- ESC 키로 모달을 닫는 기능이 없다.
- 아이콘 버튼의 접근 가능한 이름이 부족하다.
- 폼 라벨이 실제 `<label for>`로 연결되어 있지 않다.

### 9.4 반응형 리스크

- 미디어 쿼리가 없다.
- 사이드바 220px, 상세 패널 310px, 테이블이 동시에 존재해 작은 화면에서 레이아웃이 깨질 수 있다.
- 테이블 컬럼이 많아 모바일 사용성이 낮다.
- 모달은 `max-width:92vw`가 있어 어느 정도 줄어들지만 내부 2열 폼은 좁은 화면에서 불편할 수 있다.

### 9.5 유지보수 리스크

- HTML, CSS, JavaScript가 하나의 파일에 모두 들어 있다.
- 인라인 이벤트 핸들러가 많아 로직 추적과 테스트가 어렵다.
- 상단 액션 HTML이 JavaScript 문자열로 중복 정의되어 있다.
- 데이터와 UI 카운트가 분리되어 있어 불일치가 쉽게 발생한다.
- DOM ID 기반 직접 조작이 많아 구조 변경에 취약하다.

## 12. 결론

`docs/test.html`은 SalesKit의 핵심 업무 흐름을 빠르게 설명하는 정적 프로토타입이다. 딜 목록, 검색/필터/정렬, 상세 패널, 활동 기록, 제품 단가 자동 입력처럼 데모 가치가 높은 기능은 이미 들어 있다.

다만 현재 상태는 실제 서비스라기보다 화면과 상호작용을 보여주는 시안이다. 가장 큰 문제는 데이터 기반으로 계산되어야 할 카운트와 대시보드 지표가 정적 값으로 남아 있다는 점이다. 또한 등록, 수정, 복사, 내보내기, OCR, 휴지통 복구 등은 실제 기능이 아니라 alert 또는 시뮬레이션이다.

다음 단계에서는 먼저 화면 숫자와 대시보드 지표를 `DEALS`, `CONTACTS`, `COMPANIES`, `PRODUCTS`에서 계산하도록 바꾸고, 그 다음 회사/제품 검색, 내보내기, 등록 저장, 접근성 보강 순서로 개선하는 것이 적절하다.
