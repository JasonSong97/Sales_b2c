## 1. 제품 정체성

- 서비스는 **B2C 개인 영업 전용 도구**로 시작한다.
- 주 타겟은 **남성 30~50대 영업직**, 특히 40~50대 가능성이 높다.
- 제품 정체성은 하나로 좁히지 않고 다음을 모두 포함한다.
  - 개인용 경량 CRM
  - 현장 영업 카드 앱
  - 딜/파이프라인 관리 도구
  - 일정/회의록 중심 영업 업무 관리 도구
- 초기 가격 가설은 월 구독 **5,900~6,900원**이다.

## 2. 서비스명

- 한국명: **한손에 영업**
- 영문명/도메인 후보: **onehand.sales**

## 3. MVP 핵심 루프

MVP는 아래 네 가지 루프를 모두 포함한다.

- 거래처/회사/제품 등록 및 빠른 검색
- 딜 생성, 단계/가능성/금액/활동 메모 관리
- 일정 등록 및 주간 일정 보고서 출력
- 회의 내용 작성, AI 회의록 생성, 딜 연결

개발 우선순위:

1. 회사/거래처/제품
2. 딜
3. 일정
4. 회의록

## 4. 플랫폼 전략

- **웹 MVP를 먼저 만든다.**
- 핵심 검증 후 iOS/Android 앱으로 확장한다.
- 모바일 앱은 React Native + Expo로 만든다.

## 5. 인증

MVP 인증 제공자는 다음 네 가지다.

- 카카오
- 구글
- 네이버
- 애플

애플 로그인은 iOS 앱 출시와 추후 모바일 구독 결제를 고려해 필수로 포함한다.

## 6. 결제/구독

- 결제/구독은 MVP에서 제외한다.
- MVP 이후 초기 유료 운영은 **계좌이체 수동 처리**로 시작한다.
- 약 1~2개월 동안 유지보수와 고객 니즈 반영을 진행한다.
- 이후 웹과 앱 모두에 **구독형 결제**를 붙인다.
- 단건 결제는 하지 않는다.

## 7. 영업 분야

- MVP는 특정 1개 영업 분야에 고정하지 않는다.
- 여러 영업 분야가 쓸 수 있는 **범용 구조**로 만든다.
- 이후 고객 반응을 보고 특정 분야별 템플릿/기능으로 확장한다.

## 8. 데이터 모델 방향

기본 모델은 명확한 영업 도메인 구조를 사용한다.

- Company
- Contact
- Product
- Deal
- Schedule
- MeetingNote

다만 모든 주요 엔티티에 다음 확장 요소를 둘 수 있게 설계한다.

- 태그
- 메모
- 커스텀 필드
- metadata

커스텀 필드는 DB 구조에는 열어두되, MVP UI에서는 숨긴다.

## 9. 태그

태그는 주요 엔티티 전체에 붙일 수 있다.

- 회사
- 거래처(담당자)
- 제품
- 딜
- 일정
- 회의록

MVP에서 사용자는 태그를 직접 생성/수정/삭제할 수 있다.

## 10. 회사와 거래처

- `거래처`는 회사가 아니라 **회사에 소속된 사람**, 즉 Contact를 의미한다.
- 구조는 `Company -> Contact`다.
- 화면 메뉴명은 **거래처(담당자)** 로 사용한다.
- 회사 위치를 기본값으로 사용한다.
- 거래처(담당자) 위치는 선택 입력이다.
- 회사의 `분야(산업)`은 기본 목록을 제공하되, 사용자가 직접 입력할 수 있다.

## 11. 회사 로그

- 회사 로그는 회사 자체의 연혁/히스토리/변경 내역을 기록하는 용도다.
- 영업 접촉 이력이나 상담 이력이 아니다.
- 사용자가 직접 작성한다.
- 필드는 다음과 같다.
  - 날짜
  - 제목
  - 내용

예:

- 올해 이익 +100조
- 내년 전망치 +120조 예상
- 신규 공장 증설
- 조직 개편

## 12. 제품

- UI 용어는 **제품**을 사용한다.
- 코드/데이터 모델 용어는 `Product`를 사용한다.
- `상품`, `영업상품` 표현은 정리 대상이다.
- 제품 단가는 선택 입력이다.
- 제품 분류는 기본 목록을 제공하되, 사용자가 직접 입력할 수 있다.

제품은 다음과 모두 연결 가능하다.

- 회사
- 거래처(담당자)
- 딜

제품 연결에는 연결 타입을 둔다.

예:

- 관심 제품
- 기존 납품 제품
- 제안 중 제품
- 경쟁 제품
- 유지보수 대상 제품
- 기타

## 13. 딜

- 딜 금액은 필수 입력이다.
- 제품 단가가 있으면 딜 생성 시 참고값/기본값으로 불러올 수 있다.
- 딜 금액은 KRW 기본이다.
- 다른 통화는 추후 확장한다.

MVP 기본 딜 단계:

- 초기 접촉
- 협의중
- 성사
- 실패

딜 단계는 자유롭게 앞뒤 변경 가능하다. 단, 모든 변경은 딜 활동 로그에 자동 기록한다.

딜 가능성 표시는 다음 방식으로 한다.

- 기본: 긍정 / 중립 / 부정
- 고급 옵션: 숫자 퍼센트 입력

## 14. 딜 활동 로그

딜 활동 로그 필드:

- 날짜
- 타입
- 제목
- 내용
- 자동 생성 여부

기본 활동 타입:

- 메모
- 전화
- 미팅
- 이메일
- 단계변경
- 회의록연결

사용자는 직접 활동 타입을 생성할 수 있다.

## 15. 일정관리

일정관리는 MVP에 포함한다.

범위:

- 일정 CRUD
- 딜/회사/거래처 연결
- 주간 일정 보고서 출력
- 알림 기능
- 구글 캘린더 연동

구글 캘린더 연동 순서:

1. 구글 캘린더 일정을 우리 서비스로 가져오기
2. 이후 우리 서비스 일정을 구글 캘린더로 내보내기 추가 가능

양방향 동기화는 하지 않는다.

주간 일정 보고서 출력:

- 화면에서 주간표로 보기
- PDF 다운로드
- Excel 다운로드

## 16. 알림

웹 MVP 알림 채널:

- 이메일
- 브라우저 푸시

모바일 앱 출시 후:

- 모바일 앱 푸시 추가

카카오 알림톡은 MVP 알림 채널에서 제외한다.

알림 대상:

- 일정 시작 전 알림
- 딜 마감일 알림
- 다음 액션/후속 조치 알림
- 회의록 생성 완료 알림

알림 시간은 기본값을 제공하고, 사용자가 수정할 수 있게 한다.

## 17. 회의록

회의록은 MVP에 포함한다.

MVP 범위:

- 사용자가 텍스트로 회의 내용 입력
- AI 회의록 생성
- 사용자가 수정 후 저장
- 딜 연결
- 딜 활동 로그 자동 생성

MVP 제외:

- 음성 녹음
- STT

MVP 회의록 AI 결과 항목은 고정 9개다.

- 날짜
- 회사
- 담당자
- 부서
- 품목
- 진행단계
- 상세내용
- 향후계획
- 필요액션

추후 사용자가 항목/템플릿을 커스터마이즈 가능하게 확장한다.

회의록은 딜 없이도 저장 가능하며, 나중에 딜과 연결할 수 있다.

회의록 연결 방식:

- 작성 시 회사/거래처 직접 선택 가능
- AI가 회의 내용에서 회사/담당자를 추출해 후보 제안
- 딜 연결 시 딜의 회사/거래처 정보 자동 상속

## 18. 명함 OCR

명함 스캔/OCR은 MVP 필수다.

웹 MVP:

- 명함 이미지 업로드 기반 OCR

모바일 앱 출시 후:

- 카메라 촬영 기반 명함 스캔

OCR 결과는 항상 사용자가 확인/수정 후 저장한다. 신뢰도가 높아도 자동 저장하지 않는다.

명함 OCR 저장 시 회사 처리:

- 기존 회사 후보를 보여준다.
- 사용자가 기존 회사 연결 / 새 회사 생성 / 회사 없이 저장 중 선택한다.
- 자동 매칭은 후보 제안까지만 한다.

## 19. Import / Export

Export 대상:

- 회사
- 거래처(담당자)
- 제품
- 딜
- 일정
- 회의록

주간 일정 보고서도 PDF/Excel을 지원한다.

Export 시 민감 데이터:

- 기본 제외
- 사용자가 명시적으로 포함 선택 가능
- 포함 시 경고 표시

Import는 MVP에 포함한다.

Import 대상:

- 회사
- 거래처(담당자)
- 제품
- 딜

Import 방식:

- AI가 Excel/CSV 컬럼을 자동 매핑한다.
- 사용자가 매핑 결과를 확인/수정한다.
- 확정 후 Import한다.

일정/회의록 Import는 MVP 제외다.

## 20. 민감 데이터

민감 데이터 기준:

- 개인 메모
- 회의록 본문
- 딜 금액
- 사용자가 `민감`으로 표시한 모든 데이터

Admin에서는 민감 데이터가 기본 마스킹된다.

원문 조회가 필요한 경우:

- 명시적 액션 필요
- 사유 입력 필요
- 감사 로그 기록 필수

민감정보 저장 경고는 MVP에 포함한다.

- 저장 전 경고 표시
- 사용자가 확인 후 저장
- 키워드/패턴 자동 감지는 MVP에서 하지 않음

개인 참고 메모 명칭은 **개인 메모**로 한다.

`오프더레코드` 표현은 MVP UI에서 사용하지 않는다.

Admin 웹은 MVP에 최소 포함한다.

우선 필요한 Admin 기능:

- 사용자 목록/상세
- 전체 딜 조회

## 25. 모노레포 구조

모노레포 루트는 저장소 루트다.

기존 `AI_INFRA` 폴더는 한 단계 감싸는 구조였으므로 제거하고, 내부 항목을 저장소 루트로 올린다.

루트에는 `package.json`을 두지 않는다.
workspace 설정도 사용하지 않는다.

FE와 BE는 모노레포 안에 있지만 패키지/의존성은 공유하지 않는다.

최종 구조:

```text
AGENT/
  PLANNING/
  CONVENTION/
  ARCHITECTURE/
  DECISIONS/
FE/
  user-web/
  admin-web/
BE/
archive/
```

FE 하위:

- `FE/user-web`
- `FE/admin-web`

BE:

- 단일 NestJS 서버
- User API: `/api/*`
- Admin API: `/admin/api/*`

모바일 폴더는 지금 만들지 않는다. 웹 MVP 이후 모바일 개발 시 추가한다.

## 26. 테스트 자동화

MVP 테스트 자동화는 User Web과 Admin Web을 모두 포함한다.

원칙:

- User Web E2E는 `FE/user-web`에서 관리한다.
- Admin Web E2E는 `FE/admin-web`에서 관리한다.
- 루트 공용 패키지나 workspace를 만들지 않는다.
- Playwright를 각 FE 앱의 E2E 도구로 사용한다.
- 백엔드는 위험도 높은 도메인/권한/Import/민감정보 흐름을 중심으로 단위/통합 테스트를 둔다.

우선 E2E 대상:

- 로그인/보호 라우트
- 회사/거래처/제품 CRUD
- 딜 생성/단계 변경/활동 로그
- 일정 CRUD와 연결
- 회의록 저장/딜 연결
- Import/Export 핵심 흐름
- Admin 사용자 목록/상세
- Admin 전체 회사/거래처/제품/딜 조회
- Admin 민감정보 마스킹/원문 보기 사유 입력/감사 로그
