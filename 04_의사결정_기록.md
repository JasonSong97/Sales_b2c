# 의사결정 기록 (Decision Log)

> 기획/기술 선택의 사유를 정리. 나중에 "왜 이렇게 했지?"의 답을 여기서 찾는다.

---

## 1. 제품 정체성 관련

### D-001. 타겟은 영업사원 "개인" (B2C)
**결정**: 회사가 도입하는 B2B가 아니라 영업사원 개인 대상 B2C

**근거**:
- 영업사원의 개인 페인포인트 (노트북↔모바일 동기화 안 됨)는 회사가 해결해주지 않음
- B2B 영업은 사이클 길고 의사결정 복잡
- 개인이 직접 결정하는 모델이 빠른 시장 검증 가능

**영향**:
- 회사 양식 강제 X (회사 양식 기반 견적서/서류 자동 생성 제외)
- 수익화/결제는 현재 단계 범위 밖 (앱 출시 후 별도 기획)

---

### D-002. 메인 정체성은 "영업 관리"
**결정**: 캘린더, 회의 녹음, 일정 관리 같은 부가 기능은 제외

**근거**:
- 정체성이 흔들리면 사용자도 우리도 헷갈림
- "이 앱이 이거 해줘?"의 답이 명확해야 함
- 선택지를 많이 준다고 좋은 게 아님 (음식점 비유)

**영향**:
- 캘린더는 본인이 쓰던 앱 쓰게 함
- 회의 녹음 제외 (구조적으로 완성도 못 냄)
- 영업 데이터 + 거래처 + 메모에만 집중

---

### D-003. 완성도 60% 기능은 만들지 않는다
**결정**: "고깃집의 김치찌개" 원칙

**근거**:
- 부가 기능의 완성도가 낮으면 서비스 전체 신뢰도 떨어짐
- "있구나" 수준의 김치찌개는 그 집을 "그런 집"으로 만듦
- 핵심에 집중해서 80%+ 만들기

**영향**:
- 회의 녹음 (전화 끊김으로 60% 한계) → 제외
- 견적서 (회사 양식 매칭 한계) → 제외

---

### D-004. 빠른 출시 + 사용자 피드백 반복
**결정**: 4주 안에 Phase 3까지 배포 → 베타 사용자 → 피드백 → 고도화

**근거**:
- 우리 머릿속 가설보다 실제 사용자 반응이 진실
- 빨리 틀리는 게 비용 적음
- AI 활용으로 가능한 일정

**영향**:
- 테스트 코드 최소화 (인증/엑셀 Import만 예외)
- 인프라는 Railway로 빨리 시작 (1개월 후 VM 이전)
- 부가 기능은 베타 피드백 보고 결정

## 2. 아키텍처 관련

### D-101. React + Vite (Next.js 아님)
**결정**: 프론트엔드는 React + Vite

**근거**:
- SEO 불필요 (로그인 후 사용 앱)
- Next.js의 SSR/SSG 가치 낮음
- Vite 빌드 빠르고 설정 간단
- PWA 셋업 간단

**대안 검토**:
- ❌ Next.js: SSR 안 쓰면 가치 적음, 무거움
- ❌ React Native: 노트북에서 봐야 한다는 요구로 비추

---

### D-102. 모노레포 아니라 레포 분리
**결정**: `sales-frontend`, `sales-backend` 별도 Git repo, `SALES_WORKSPACE`로 로컬 묶음

**근거**:
- 2인 팀 + AI 활용에서 레포 분리가 컨텍스트 명확
- Vercel은 1레포 1프로젝트가 자연스러움
- 백엔드는 VM 단독이라 모노레포 이점 적음
- Turborepo/Nx 학습 비용 절약

**영향**:
- FE/BE 타입 공유는 `AGENT/api-contract.md` 수동 정의로 시작
- 나중에 OpenAPI 자동 생성으로 전환 가능

---

### D-103. Railway 먼저 → 1개월 후 VM 이전
**결정**: 백엔드 호스팅을 Railway로 시작, 1개월 후 VM(Lightsail Seoul $10)로 이전

**근거 (Railway 먼저)**:
- Day 1~2를 인프라 셋업이 아닌 기능 개발에 투입 가능
- 셋업 시간: Railway 2시간 vs VM 7~8시간
- 4주 출시 목표에 인프라 시간 잠식 방지

**근거 (1개월 후 VM 이전)**:
- 사용자 검증 후 비용/성능 최적화
- 한국 리전(Seoul)으로 레이턴시 개선
- 비용 고정 ($10/월), 트래픽 증가에도 안정
- 풀 컨트롤 + 부가 작업 (Cron, OCR 등) 자유

**대안 검토**:
- ❌ 처음부터 VM: 출시 늦어짐
- ❌ Railway 계속: 한국 리전 없어 레이턴시 +50ms, 트래픽 늘면 비용 ↑
- ⚠️ Fly.io: 한국 리전(도쿄) 좋지만 학습 곡선

**이전 조건**:
- 월 비용 $30 초과 OR
- 사용자 100명+ OR
- Cold start 불만 누적

---

### D-104. ORM은 Prisma
**결정**: Prisma 사용 (Drizzle/TypeORM 아님)

**근거**:
- 마이그레이션 도구 압도적
- AI가 가장 잘 작성하는 ORM
- 타입 안전성 최상
- Nest와 통합 사례 많음

**대안 검토**:
- ❌ Drizzle: SQL-first 좋지만 마이그레이션 DX 약함
- ❌ TypeORM: 최근 관리 강도 약화

---

### D-105. 인증: Supabase Auth + Nest 검증 하이브리드
**결정**: 회원가입/로그인은 Supabase Auth, 권한/Role 검증은 Nest

**근거**:
- Supabase Auth: 이메일 인증, 소셜 로그인 자동 (카카오/구글 30분 추가)
- Nest: Role 기반 권한 (USER/ADMIN), 비즈니스 로직
- 둘 다의 장점 취함

**대안 검토**:
- ❌ Nest JWT 자체 구현: refresh 회전/이메일 인증/소셜 로그인 다 직접 짜야 함
- ❌ Supabase Auth만: Role/권한 로직이 BE에 일관적이지 않음

---

### D-106. 파일 업로드는 signed URL 패턴
**결정**: 클라이언트가 Supabase Storage에 직접 업로드 (백엔드 경유 X)

**근거**:
- 백엔드 트래픽/메모리 절약
- 작은 VM(2GB)에서도 대용량 파일 OK
- Supabase 권장 패턴

**흐름**:
1. FE → BE: 업로드 요청 (메타데이터)
2. BE → FE: signed URL 발급
3. FE → Supabase Storage: 직접 PUT
4. FE → BE: 완료 알림
5. BE → DB: 경로 저장

---

### D-107. 모바일은 React Native (Expo)
**결정**: React Native + Expo (Managed Workflow) + EAS Build로 iOS/Android 동시 개발·배포

**근거**:
- App Store / Google Play 정식 배포 필요 (비즈니스 신뢰도)
- iOS 푸시 알림 (PWA 푸시는 iOS 제한 큼)
- 카메라 UX (명함 스캔이 핵심) → 네이티브 카메라가 우수
- 오프라인 저장 (SQLite) 안정성
- 사용자가 "진짜 앱"으로 인식
- React 지식 재사용 (TypeScript, TanStack Query, React Hook Form, Zod 그대로) vs Flutter는 Dart 신규 학습
- Expo Managed + EAS Build로 macOS/Xcode 셋업 부담 없이 1일 내 빌드/배포 가능

**대안 검토**:
- ❌ **PWA**: iOS 푸시 제한, App Store/Play Store 배포 불가, 카메라 UX가 네이티브에 못 미침 → 핵심 요구(명함 스캔/푸시/스토어) 충족 불가
- ❌ **Capacitor**: 웹뷰 기반이라 카메라/오프라인 UX가 네이티브만 못함, 명함 스캔 품질이 결정적
- ❌ **Flutter**: Dart를 새로 배워야 함, 기존 React 자산(쿼리/폼/스키마) 재사용 불가

**변경 이력**:
- 초안: PWA → 2026-05-21 React Native(Expo)로 변경. 스토어 배포·푸시·카메라 UX 요구로 네이티브 결정

## 3. 개발 방식 관련

### D-201. 세로 슬라이스 개발 강제
**결정**: 한 기능을 DB→BE→FE→배포→QA까지 끝내고 다음으로

**근거**:
- 끝까지 동작 확인할 때까지 빠른 피드백
- 중간 설계 오류 빨리 발견
- 2인 팀이 BE/FE 각각 만들면 통합 단계에서 충돌 많음

**예외**:
- 인프라 셋업 단계(Day 1~2)는 수평적 작업 OK

### D-202. 테스트 코드 거의 없음
**결정**: 자동화 테스트 최소, 수동 QA 위주

**근거**:
- 빠른 출시 우선
- 4주 일정에 테스트 작성 시간 빠짐
- 베타 단계라 회귀 영향 작음

**예외 (반드시 테스트)**:
1. **인증/권한 로직**: 다른 사용자 데이터 노출 위험
2. **엑셀 Import 파싱**: 사용자 엑셀 형식 천차만별, 회귀 잦음

### D-203. 도메인은 Day 1에 구매
**결정**: 로컬 개발 시작 전에 도메인부터

**근거**:

### D-204. 락인 회피 5원칙 (Day 1부터 적용)
**결정**: Railway에 락인되지 않게 코드 작성

### D-205. 코드 스타일·주석·로깅은 빅테크 표준을 따른다
**결정**: 4 클라이언트 + 백엔드 전반에 Google/Meta/Microsoft/Amazon/Netflix가 공유하는 표준을 적용한다.

**주석 표준 (4 폴더 공통)**:
- 기본은 주석 없음. WHY가 비명백할 때만 한 줄
- WHAT 설명 금지 (코드/타입/명명이 함)
- JSDoc은 public API 진입점만, 타입 중복 금지
- 함수 본문에 `// 1. ... // 2. ...` 단계 주석 금지 → 함수를 더 작게 쪼갠다
- 한국어 코드 번역 주석 금지
- 주석 처리된 코드 금지 (Git이 보관)
- 변경 이력 주석 금지 (`git log`/`git blame`이 함)
- TODO/FIXME/HACK에는 이슈 번호 또는 날짜 강제

**로깅 표준 (백엔드)**:
- 구조화 JSON (pino) — 짧은 영문 키 + 컨텍스트 객체
- WHEN/WHO/WHERE는 인터셉터에서 자동 주입 — 코드에 손으로 적지 않음
- ASCII 박스 / 다단 구분선 / 문장형 메시지에 변수 박기 금지
- 도메인 코드는 throw만, 로그는 Exception Filter가 생성
- PII는 pino `redact` 옵션으로 자동 마스킹
- 로그 레벨: fatal > error > warn > info > debug (의미별로 정확히)
- `console.log` 금지

**로깅 표준 (클라이언트 — 웹/모바일/Admin)**:
- 브라우저/RN에서는 Sentry SDK가 표준 채널
- 사용자 행동 이벤트는 별도 분석 도구로 (Mixpanel/PostHog 등 — 채택 시점은 [09_PMF_검증.md](09_PMF_검증.md) 6.4)
- 모바일은 오프라인에서 발생한 로그를 큐에 보관 후 동기화 시 일괄 전송
- `console.log` 금지 — `logger` 래퍼만 사용 (production에서 자동 비활성)

**근거**:
- 로그는 사람이 grep 하는 게 아니라 Sentry/Datadog/CloudWatch가 파싱·집계·알람한다
- 빅테크 표준은 "수동 추적성" 대신 "도구가 자동 추적" 방향. 코드 노이즈 감소 + 일관성 확보
- AI에게 일관된 출력을 강제하는 것이 작은 팀에서 가장 큰 ROI
- 4 클라이언트가 같은 로그 키 컨벤션을 쓰면 한 사용자의 행동을 traceId로 풀스택 추적 가능

**대안 검토**:
- ❌ 한국 SI 스타일 (ASCII 박스 로그, 모든 메서드 docstring, `_validation_*_by` 같은 명명): 코드 노이즈, drift 위험, 도구 통합 불가
- ❌ "주석 자유롭게": 팀원/AI별 일관성 깨짐. 리뷰 비용 증가
- ✅ 빅테크 표준: 검증된 공개 가이드, 도구·생태계 정렬

**적용 위치**:
- 백엔드: [AGENT/Backend/comment-rules.md](AGENT/Backend/comment-rules.md) 8장 (구조화 로깅)
- 프론트: [AGENT/Frontend/comment-rules.md](AGENT/Frontend/comment-rules.md)
- 모바일: [AGENT/Mobile/comment-rules.md](AGENT/Mobile/comment-rules.md)
- Admin: [AGENT/Admin/comment-rules.md](AGENT/Admin/comment-rules.md)

**위반 시**: PR 리뷰에서 거부. AI가 위반한 출력은 사용자가 반려.

## 4. 보류/제외 결정

### D-301. 견적서 자동 생성 → 제외
**사유**: 회사 양식이 다양해서 60% 이상 맞추기 어려움. 차라리 안 만드는 게 신뢰도 보존.

### D-302. 회의 녹음 → 제외
**사유**: 전화 끊김으로 구조적 완성도 한계. 통화 끊길 때마다 수동으로 재시작해야 함.

### D-303. 자체 캘린더/일정 관리 → 보류
**사유**: 본인이 쓰던 캘린더 앱 쓰는 게 빠름. 정체성 흐림. 사용자 요청 누적되면 재검토.

### D-304. 회사 양식 기반 자동 서류 생성 → 제외
**사유**: 회사 양식 매칭 불가능. 견적서와 동일 이유. (사용자 결제/송금 시스템과는 무관)

### D-305. 상품 카탈로그 자동 입력 → Phase 5
**사유**: 좋은 기능이지만 핵심 후순위. Phase 1~3 마무리 후 베타 피드백 보고 결정.
