# 추가 권장 작업 (Recommendation)

> 통화·기획서에는 명시되지 않았지만, 실제 영업 관리 서비스를 운영하기 위해 **추가로 검토할 가치가 있는** 작업을 정리.
>
> **이 문서는 "결정 사항"이 아니라 "검토 후보"**. 사용자가 직접 보고 채택/거절/연기를 결정한다.
>
> **원칙 준수**:
> - "고깃집의 김치찌개" 원칙 — 부가 기능은 완성도 80%+ 가능할 때만
> - 메인 정체성("영업 관리")을 흐리는 항목은 [선택지 많다고 좋은 게 아님] 원칙으로 배제 후보
> - DB 스키마는 의도적으로 언급하지 않음 (사용자가 전체 그림 보고 직접 설계 예정)
>
> 작성일: 2026-05-21

---

## 분류 기준

| 계층 | 정의 | 권장 시점 |
|------|------|----------|
| **🔴 핵심 (Core)** | 출시 시점에 없으면 서비스 신뢰도/안정성이 무너지는 항목. 통화엔 안 나왔지만 운영상 필수 | Phase 0~3 (4주 내) |
| **🟡 확장 (Extension)** | 베타 검증 후 핵심 가치를 강화하는 항목. 정체성 일치 + 완성도 가능 | Phase 4 (베타 후) |
| **🟢 브레인스토밍 (Ideation)** | 가능성은 있지만 정체성 흐림 위험 또는 검증 필요. 일단 적어만 둠 | 미정 / backlog |

## 🔴 핵심 (Core) — 출시 전 반드시 결정

### C-01. 데이터 영구 삭제 vs Soft Delete
**무엇**: 거래처/영업건/메모 삭제 시 DB에서 즉시 제거 vs `deleted_at` 컬럼으로 표시

**왜**:
- 영업 데이터는 실수로 지워지면 매출 직결 (한 거래처가 수개월간의 영업 기록을 포함)
- "삭제했는데 복구해줘" 요청은 100% 발생
- 회사가 아닌 개인 사용자라 백업 책임이 사용자에게 없음 → 서비스가 보호 책임

**검토 사항**:
- 모든 도메인 일괄 Soft Delete? 아니면 거래처/영업건만?
- 휴지통 UI 제공? 자동 영구삭제 기간(30일/90일)?
- 명함 OCR 원본 이미지는 어떻게? (저장 비용 vs 재OCR 가능성)

---

### C-02. 계정 삭제 / 데이터 내보내기 (법적 의무)
**무엇**: 회원 탈퇴 기능 + 본인 데이터 전체 다운로드(JSON/엑셀)

**왜**:
- 한국 개인정보보호법 + 위치정보법: 본인 데이터 열람·삭제권 의무
- B2C 서비스는 회사 정책 면피 불가
- 앱스토어 심사 통과 조건 (Apple은 2022년부터 "Sign in with Apple"이 있으면 계정 삭제 기능 강제)

**검토 사항**:
- 탈퇴 즉시 삭제 vs 7일 유예
- 데이터 내보내기 포맷 (엑셀 = 사용자 친화, JSON = 완전성)
- OCR 이미지/명함 원본도 같이 내려줄지

---

### C-03. 입력 데이터 검증 + 정규화 (전화번호/이메일/날짜)
**무엇**: 거래처 등록 시 전화번호 형식 자동 변환, 이메일 정규식 검증, 중복 체크

**왜**:
- 영업사원은 모바일에서 빠르게 입력 → 오타/형식 오류 빈번
- "010-1234-5678" / "01012345678" / "+82 10-1234-5678" 같은 거 다 들어옴
- 나중에 필터링/검색/엑셀 Import 시 일관성 없으면 작동 안 함
- 한국 휴대폰 정규화, 회사 전화/팩스 구분 정도는 출시 전 필수

**검토 사항**:
- 통일된 저장 형식 (E.164 권장: `+821012345678`)
- 표시 형식 (한국: `010-1234-5678`)
- 같은 전화번호의 거래처 중복 등록 차단 여부

---

### C-04. 검색 (Full-text)
**무엇**: 통화에서 "송재근 치면 어디 누구 송재근 나옴"이라고 했는데, 이게 **컬럼 필터(2.3)와는 별개의 검색**임

**왜**:
- 통화에서 명시적으로 요구됐음에도 [01_서비스_기획서.md](01_서비스_기획서.md)와 [03_개발_전략_및_일정.md](03_개발_전략_및_일정.md)에 **"검색" 단독 항목이 누락**됨 (Phase 1 목록에 "검색" 한 줄 있긴 함)
- 컬럼 필터는 "서울 + 영업중"처럼 조건 조합, 검색은 "송재근"처럼 자유 텍스트
- 이름/회사명/메모/명함 OCR 텍스트까지 통합 검색되어야 가치 발생

**검토 사항**:
- PostgreSQL `pg_trgm` + `gin` 인덱스 vs Meilisearch 별도 도입
- 한국어 검색 품질 (조사 처리, 부분 일치)
- 검색 범위 토글 (거래처만 / 메모까지)

---

### C-05. 모바일 자동 로그인 + 토큰 만료 처리
**무엇**: 앱 재실행 시 자동 로그인, 토큰 만료 시 무중단 갱신

**왜**:
- 영업사원은 하루에도 앱을 수십 번 열고 닫음 → 매번 로그인 시키면 즉시 사용 중단
- expo-secure-store에 refresh token 보관 + 401 시 자동 재시도는 출시 시점 필수
- 통화엔 "로그인/회원가입"만 있고 토큰 라이프사이클은 누락

**검토 사항**:
- access token 수명 (1h?) / refresh token 수명 (30d?)
- refresh 회전(rotation) 여부
- 동시 만료 요청 처리 (race condition)

---

### C-06. 오프라인 → 온라인 동기화 충돌 UX
**무엇**: [02_기술스택_및_아키텍처.md](02_기술스택_및_아키텍처.md) 3.4에 "last-write-wins"라고 적혀 있지만, **사용자에게 보이는 UI 정책은 미정**

**왜**:
- 노트북에서 거래처 메모 수정 + 동시에 핸드폰에서 다른 내용으로 수정 → 둘 중 하나 사라짐
- 영업사원은 "내가 적은 게 왜 사라졌지?"가 핵심 신뢰 이슈
- 메모는 특히 위험 (덮어쓰기 = 분 단위 작업 손실)

**검토 사항**:
- 메모는 append-only로 (덮어쓰기 대신 새 메모 추가)
- 충돌 시 토스트 알림 + 휴지통에서 이전 버전 복구 가능
- 또는 필드별 last-write-wins (이름/전화는 덮어씀 OK, 메모는 보존)

---

### C-07. 명함 OCR 실패/저품질 처리
**무엇**: CLOVA OCR이 인식 실패하거나 낮은 신뢰도로 반환했을 때 UX

**왜**:
- 실제 명함은 디자인 천차만별 (영문/세로조판/금박 등)
- "OCR 실패했어요" 알림만 띄우면 사용자는 "이 앱 안 됨" 결론
- 통화에서 "명함 OCR이 핵심"이라고 강조됐지만 실패 케이스 정책은 없음

**검토 사항**:
- 신뢰도 임계값 (0.7 미만은 사용자 수동 입력 화면으로 fallback)
- 인식 실패 시 이미지만 저장 + 거래처 카드에 "OCR 대기" 상태
- 실패 명함을 모아 재시도 큐 (CLOVA 모델 업데이트 시 일괄 재처리)
- 영문 명함 / 일본어 명함 처리 (CLOVA는 한국어 특화)

---

### C-08. 에러 메시지 / 토스트 일관성
**무엇**: 모든 클라이언트(User웹/모바일/Admin)에서 에러를 같은 톤·UX로 표시

**왜**:
- 백엔드는 `code: 'CUSTOMER_NOT_FOUND'` 같은 코드 반환 ([Backend/architecture.md](AGENT/Backend/architecture.md) 6.2)인데 클라이언트별로 어떻게 풀어쓸지 표준 없음
- "오류가 발생했습니다" 같은 무성의 메시지는 사용자 이탈 직결
- AI에게 에러 메시지 작성 맡기면 매번 다른 톤으로 만듦

**검토 사항**:
- 에러 코드 ↔ 사용자 메시지 매핑표 (`AGENT/error-messages.md` 신규)
- 토스트 라이브러리 표준 (`sonner` 등)
- 복구 액션 버튼 ("다시 시도", "고객센터" 등)

---

### C-09. 로그/모니터링 최소 셋
**무엇**: Sentry 외에 어떤 로그를 남길지 표준화

**왜**:
- "분명 거래처 등록했는데 안 보여요" → 사용자 행동 로그 없으면 원인 추적 불가
- [03_개발_전략_및_일정.md](03_개발_전략_및_일정.md) Day 6~7에 "Sentry 셋업"만 있고 행동 로그 정책 없음
- 베타 단계의 사용자 피드백 검증에 필수

**검토 사항**:
- 어떤 액션을 기록할지 (로그인, CRUD, OCR 호출, 동기화 충돌)
- PII(이름/전화) 마스킹 정책
- 로그 보관 기간 (Sentry/Supabase 무료 한도)
- 모바일에서 발생한 오프라인 에러도 동기화 시 함께 전송

---

### C-10. 베타 사용자 식별 / 피드백 채널 인앱 통합
**무엇**: [03_개발_전략_및_일정.md](03_개발_전략_및_일정.md) Week 4에 "디스코드 또는 카톡방"이라고 적혀 있지만, 인앱 피드백 버튼이 없으면 의견 수렴 안 됨

**왜**:
- 영업사원이 "운전 중 안 보이네" 발견해도 카톡방까지 가는 마찰이 큼
- 베타 단계에서 가장 큰 자산은 사용자 음성 — 수집 비용을 0에 가깝게
- 베타가 끝나면 자연 소멸하는 채널 (영구 채널 X)

**검토 사항**:
- 설정 화면 > "의견 보내기" 버튼 (텍스트 + 스크린샷 자동 첨부)
- 백엔드에 저장 vs Slack/디스코드 웹훅으로 바로 전송
- 베타 사용자 식별 플래그 (`users.is_beta`)

## 🟡 확장 (Extension) — 베타 검증 후 정체성 안에서 강화

### E-01. 거래처 그룹/태그
**무엇**: 거래처에 자유 태그 (`#VIP`, `#잠재고객`, `#반도체` 등)

**왜**:
- 컬럼 필터(지역/상태)는 이미 있지만, **사용자 정의 분류**는 없음
- 영업사원마다 자기 방식의 분류가 있음 (예: 친밀도, 거래 빈도, 산업군)
- "고깃집의 김치찌개" 원칙 통과: 단순 기능이라 완성도 100% 가능

**왜 확장(Phase 4)에 두는가**:
- 핵심 4주에 넣으면 컬럼 필터/탭 필터와 UX 충돌 가능성
- 베타 사용자가 실제로 "분류" 요청하는지 검증 후 도입

---

### E-02. 최근 본 거래처 / 즐겨찾기
**무엇**: 자주 보는 거래처 빠른 접근

**왜**:
- 활성 영업 건은 보통 5~10개 거래처에 집중
- 매번 검색/필터 거치면 마찰
- "운전 중 빠른 조회" 페인포인트(통화 원본)의 보강

---

### E-03. 영업건 단계별 진행 통계 (개인 대시보드)
**무엇**: 본인의 lead/in_progress/won/lost 카운트, 월별 won 금액 추이

**왜**:
- 영업사원의 자기 진단 도구 — 회사 양식과 무관한 본인 KPI
- 정체성 일치: "회사가 도입하지 않는 개인용 영업 관리"
- Admin의 통계와 별개로 본인용 차트
- 단, 통화에선 직접 요구되지 않음 → 베타에서 실제 수요 확인 후

---

### E-04. 명함 이미지 보관 + 원본 보기
**무엇**: OCR 후 원본 명함 이미지를 거래처 상세에서 다시 볼 수 있게

**왜**:
- OCR이 놓친 정보(손글씨 메모, 명함 뒷면)를 사람이 보고 확인
- "이 명함 그때 어디서 받았더라" 같은 시각 기억 단서
- 이미 Supabase Storage에 저장될 예정이라 추가 비용 거의 없음

---

### E-05. 엑셀 Import 컬럼 자동 매핑 학습
**무엇**: 사용자가 매번 컬럼 매핑하지 않도록 저장/재사용

**왜**:
- [03_개발_전략_및_일정.md](03_개발_전략_및_일정.md) Day 17~18에 "컬럼 매핑 UI"가 있는데 매번 새로 하면 짜증
- 같은 사용자는 보통 같은 엑셀 양식을 반복 사용
- 매핑 프리셋 저장 + 자동 적용

---

### E-06. 메모/거래처 변경 이력
**무엇**: "이 거래처 정보를 언제, 무엇으로 바꿨는지" 타임라인

**왜**:
- "분명 010-1111이었는데 누가 바꿨지?" 자기 의심 해소
- C-06 충돌 정책과 연결 — 충돌 발생 시 이력에서 복구
- audit log와 비슷하지만 사용자 본인용

---

### E-07. PDF 출력 템플릿 (월별 보고용)
**무엇**: [01_서비스_기획서.md](01_서비스_기획서.md) 시나리오 4 "월말 보고용 PDF"를 더 다듬은 형태

**왜**:
- 현재 계획은 단순 PDF Export
- 영업사원이 회사에 제출하기 좋은 포맷 (요약 + 차트)
- "회사 양식과 매칭 안 함" 원칙 유지하되, **본인 정리용 표준 템플릿** 1~2개 제공

---

### E-08. 데이터 자동 백업 다운로드 (월 1회)
**무엇**: 매월 1일 본인 데이터 전체를 엑셀로 묶어 다운로드 링크 메일 발송

**왜**:
- 사용자 = 개인 → 회사 백업 정책 없음
- 서비스 장애/계정 분쟁 시 사용자 본인이 보유한 사본이 신뢰 자산
- 자동화되면 "이 서비스가 내 데이터를 책임진다"는 인상

## 🟢 브레인스토밍 (Ideation) — 정체성 검토 필요

### I-01. 통화 자동 기록 (전화 수신/발신 로그 → 거래처 연결)
**왜 매력적**: 영업사원의 통화 = 일의 본질. 자동 로그가 있으면 "이 사람과 언제 통화했는지" 자동 정리.

**왜 위험**: 
- iOS는 통화 로그 접근 자체가 제한 (Android만 가능 = 반쪽 기능)
- "회의 녹음 → 60% 한계 → 제외"와 같은 문제
- "고깃집 김치찌개" 통과 어려움

**판단**: 60% 가능성, 제외 후보

---

### I-02. 거래처 위치 지도 표시
**왜 매력적**: 외근 동선 짤 때 유용. 지역 필터의 시각적 확장.

**왜 위험**:
- 주소 정확도 (명함의 주소가 본사? 지점? 자택?)
- 카카오맵 API 비용/약관
- 정체성 검토: "지도가 이 서비스의 핵심인가?" 아님

**판단**: 베타에서 실제 요청 누적되면 검토

---

### I-03. 거래처 생일/기념일 알림
**왜 매력적**: 영업사원의 인간관계 관리 도구.

**왜 위험**:
- 캘린더 기능과 경계 모호 → 정체성 흔들림 ([04_의사결정_기록.md](04_의사결정_기록.md) D-303과 충돌)
- 명함에 생일 적혀 있는 경우 거의 없음 → 수동 입력 필요 → 입력 부담

**판단**: 자체 캘린더 만들지 않는다는 원칙과 충돌, 보류

---

### I-04. AI 메모 요약 / 영업 인사이트 추출
**왜 매력적**: off-the-record 메모가 쌓이면 AI가 "이 거래처는 의사결정자 OOO"처럼 요약 가능.

**왜 위험**:
- 메모는 사적/민감 정보 — AI에 보내는 것 자체 거부감
- LLM 비용 + 응답 시간
- "AI 영업 비서" 같은 정체성 확장 = 핵심 메시지 흐림

**판단**: 별도 SaaS 영역. 본 서비스 안에서는 No.

---

### I-05. 회사 정보 자동 보강 (사업자번호 → 회사명/주소/업종)
**왜 매력적**: 사업자번호만 입력하면 국세청 API로 회사 정보 자동 채움.

**왜 위험**:
- 영업 대상이 사업자 번호를 다 알지는 못함
- 국세청 진위확인 API는 무료지만 회사 정보 풀 응답은 유료 API 영역
- 명함 OCR로 회사명은 이미 들어옴 — 중복 효익

**판단**: Phase 5+ 후보. 우선순위 낮음.

---

### I-06. 다중 사용자 협업 (팀 모드)
**왜 매력적**: "회사 팀이 같이 쓰는 영업 노트" 같은 B2B 확장.

**왜 위험**:
- **B2C 정체성 정면 위배** ([04_의사결정_기록.md](04_의사결정_기록.md) D-001)
- 권한/공유/충돌 처리 복잡도 폭발
- 사용자가 만든 메모를 동료에게 보여줄지 (off-the-record의 의미 상실)

**판단**: 본 서비스의 정체성과 정반대. 별도 제품으로 분기 필요 시에만.

---

### I-07. 음성 메모 → 텍스트 변환 (운전 중 입력)
**왜 매력적**: 통화 페인포인트 "운전 중"과 자연스럽게 연결. 손 안 쓰고 메모.

**왜 위험**:
- 한국어 STT 품질 (영업 용어/회사명 인식률)
- 운전 중 사용을 권장하는 모양새 (법적/책임)
- 기능 자체는 OS 받아쓰기로도 가능 ("받아쓰기 쓰세요"가 더 나음)

**판단**: OS 받아쓰기로 충분. 자체 구현 불필요.

---

### I-08. Slack/메신저 연동 (영업건 상태 변경 시 알림)
**왜 매력적**: 개인 워크플로 자동화 (Zapier 같은).

**왜 위험**:
- 개인 사용자가 본인에게 Slack 알림 보내는 시나리오는 드묾
- 회사 Slack 연결 = B2B 영역
- 정체성 흐림

**판단**: 푸시 알림(P-003)으로 충분. 별도 연동 X.
